# 七條公理：論證與反例

每條公理都來自真實失敗。引用時不要只搬結論——把「為什麼」一起帶給目標工程，
因為公理的說服力在事故敘事裡，不在祈使句裡。

## 1. 機關不靠自覺（Mechanism over Discipline）

**陳述**：任何「靠 AI/人自覺遵守」的規則都是待修的 bug，必須有 deny/fail 的
機器落點（hook deny / 腳本 exit 1 / CI fail / ratchet）。警告不算機關。

**論證**：agent 的行為分佈有長尾——90% 的 session 會遵守文檔規則，但 harness
的存在意義正是剩下 10%：上下文被壓縮後規則掉出視野、任務壓力下「先跳過測試」、
多輪對話後把警告當背景噪音。文檔規則對這 10% 無效，而事故全部發生在這 10% 裡。

**真實反例**：某工程同一天內連續兩次「靠自覺失敗」——(a) 重寫兩個 skill + 改
hook 機關，全程沒有觸發任何自動 review，靠用戶人肉指出才補做，補做的 review
真的抓到一個 gate 繞過漏洞；(b) skill 的 .md 編輯不武裝驗證 ledger，帶著紅測試
commit 而無人發現。這兩次失敗直接催生了兩個新 gate（review 證據 gate、.md 編輯
武裝驗證）。**教訓：連「建 harness 的人」都會靠自覺失敗，何況日常 session。**

**warn 的正確用法**：warn 不是機關的終態，是收數據的過渡態。上 warn 時就要寫明
升級條件：「V1 warn + 記帳，累積一週誤報數據後升 deny」。沒有升級路徑的 warn
會永遠停在 warn（見 pitfalls.md「閘後全軟」）。

## 2. 每個輸出都要有消費者（Closed Loop = No Orphan Outputs）

**陳述**：設計任何量測/記錄/報告機關時，先回答「它的輸出誰在哪個時機消費、
消費後產生什麼動作」。回答不了就不要建——沒有消費者的儀表是自我安慰。

**論證**：閉環斷點幾乎都發生在「產出→消費」的交接處，而不是產出端。收集容易
（加個 hook 就有數據），消費難（需要例行時機 + 標準化動作 + 吞吐保證）。

**檢驗方法**：對每個機關寫一行「消費鏈」：
`Stop 反思 gate → inbox 條目 → SessionStart 檢查 status:new 條目數注入提醒 → harness-iterate skill 標準化消費 → 條目改 status: converted(<commit>)`。
鏈上任何一站寫不出具體檔案/時機/動作，就是斷點。

**配套紀律**：消費端吞吐必須被觀測。收集端一天能進 5 條，消費端一拍只出 1 條，
積壓本身要上儀表板——否則 inbox 變成寫了就忘的日記。

**制度化形式：帳本註冊表**。在頂層架構文檔維護一張表：每本帳（inbox、債務
ledger、工單、偏差、事件帳本、週報…）一行，列出寫入端 / 消費端 / 節奏 / 狀態。
架構規則：(a) 新帳本上線必須同時登記消費端，沒有消費端的帳本不允許上線；
(b) 「✓ 閉環」狀態只授予消費端**有機器落點**（週報段/hook 提醒/ratchet）的帳本，
消費端只是散文協議的標 ⚠ 並列入缺口地圖；(c) 消費端優先掛在既有時機點
（SessionStart 提醒、週報段落），不另造輪子。

## 3. Ratchet 只進不退

**陳述**：債務數只許降、防護清單（關鍵路徑 manifest、白名單檢查）只許長。
任何放鬆走顯式逃生（環境變數 + 原因字串）並自動落帳本。

**論證**：harness 的退化是漸進且無聲的——一次「暫時 skip」、一次「這次先不跑」，
每次都有當下合理的理由，累積起來就是體系瓦解。ratchet 把退化變成必須顯式宣告
的動作，帳本把宣告變成週報上的可審計負債。

**實現要點**：
- 逃生必須「按次」不「按批」：skip 一次 commit 不清除 pending 狀態，下次還攔。
- 逃生自動記帳，不依賴使用者自報：`FORCE=1` 的路徑本身寫入事件帳本。
- ledger 的 schema 由 ratchet 腳本強制：新增債務必須帶 class/reason/registered/
  debt_doc，且「新增」只允許發生在登記當日（防止悄悄補登舊帳）。

## 4. 證據長存，SSOT 進 repo

**陳述**：帳本、inbox、債務 ledger、週報、設計文檔——一律放 repo 內走 git。
單機工具（本地記憶插件、`~/` 下的狀態檔）只能當快取或原始流水。

**論證**：harness 要跨機器、跨平台（多個 agent 工具共用）、跨時間（半年後考古）
生效。任何依託單機的狀態都會在換機器/換工具時靜默丟失，而 harness 的價值
恰恰在長期積累。

**實現要點**：本機事件流水（如 `~/.claude/gate-events.jsonl`）允許存在，但
聚合結果必須落 repo（週報產出時把本機計數聚合進 repo 內的 health report），
**聚合值才是可信數據**。

## 5. Harness 對可維護行為吃 TDD

**陳述**：先分類工作。reusable code/parser/validator/generator/installer/recovery、
可重現 regression fix、hook/gate/policy enforcement、release artifact、stable
contract 與危險 write path 等可維護行為變更，先寫失敗測試，再最小實作。
探索、研究、唯讀查證、設計規劃、純文檔、證據整理、compiler diagnostics、
throwaway probe 與行為不變的結構搬移不強求 TDD；一旦開始改變可維護行為就重新
分類。驗證獨立於 TDD，兩類工作都要用與風險相稱的證據收環。機關腳本的測試密度
仍應不低於業務代碼——參考實現的 hook 測試檔（1845 行）比本體（1740 行）還大。

**兩條特有紀律**：
- **真實樣本 fixture**：修誤射/漏攔時，fixture 必須用真實發生過的命令/路徑
  （去事故記錄裡找），不要造理想化樣本。理想化樣本測不出真實世界的引號嵌套、
  heredoc、複合命令。
- **正向對照**：每個「不再誤觸發」的修正，必須配一個「該觸發的仍然觸發」測試。
  真實教訓：一次降噪修正差點把部署逃生路徑（`DEPLOY_FORCE=1`）的偵測一起殺掉——
  只測「誤射消失」不測「真陽性保留」的降噪是在拆 gate。

**一鍵可信集**：`test-harness.sh` 模式——一條命令 = 全部機關測試套件 + live
一致性檢查 + ledger ratchet。改任何機關後的收環驗證就是跑這一條，沒有「只跑
我改的那個測試」的空間。

## 6. 當下歸因（Attribution at the Moment）

**陳述**：可信集裡任何測試從綠變紅，改動者必須在當下三選一，禁止留紅：
1. **改壞了** → 修代碼回綠。
2. **過時了** → 當場改測試，commit message 寫明「測試語義變更：舊預期 → 新預期，原因」。
3. **當下處理不了** → 登記進債務 ledger（pattern/class/reason/registered/debt_doc），
   登記過的紅才允許存在。

**論證**：只有改動者在改動當下知道語義變了沒有。半個月後回來，「過時 vs 壞掉」
變成考古題，考古成本十倍以上且經常無解。紅測試一旦堆積，所有測試訊號都被忽略
——這是測試體系死亡的標準路徑。

**推廣**：同樣邏輯適用於需求偏差（被糾偏的 AI 在當輪立即登記偏差 + 歸因枚舉）
與 gate 繞過（bypass 當場落帳本）。共同原理：**歸因必須發生在資訊最全的時刻**。

## 7. 小步 + 成本有界

**陳述**：harness 的自我完善每一拍只做一個最小單元（一條 inbox / 一筆債務 /
一個 gate），用便宜模型，跑不完留給下一拍。

**論證**：harness 工作是無限的（永遠有下一個可以加固的點），如果單拍成本無界，
它會跟業務開發搶資源然後被砍掉。「慢但不停」的可持續性由兩個機制保證：
ratchet 使進度不可逆（慢沒關係）；量測驅動的 priority queue 使每一拍都花在
當前最值得的單元上（不浪費）。

**反模式**：一次性大掃除（「這週末把 22 筆債務全清了」）——大掃除需要整塊
時間、整塊時間永遠等不到、等到了也會在中途發現複雜度超估而爛尾。
