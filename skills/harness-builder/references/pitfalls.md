# 設計陷阱：全部來自真實事故

動手寫任何偵測邏輯前讀一遍。每條都是某個生產 harness 真實踩過、
用回歸測試釘死的坑。

## 1. 提及 ≠ 執行

**事故**：一天之內三次誤射——grep 部署腳本的內容觸發部署警告（並非部署）、
grep 輸出含某路徑字串觸發對應 skill 注入、跑測試時測試名含 "regression" 字樣
觸發回歸測試 skill 注入。

**根因**：對 command 文本做關鍵字比對，不區分「引用這段文字」與「執行這個動作」。

**修法**：三條防線疊加——
- **唯讀命令豁免**：grep/cat/sed -n/ls/head/tail/wc 開頭的命令跳過關鍵字比對；
- **引號遮罩**：比對前把引號內文字遮掉（echo "deploy" 不是 deploy）；
- **執行位置錨定**：部署類偵測只認「行首或 && / ; 之後」的命令位置，
  出現在參數/字串裡不算。

每個防線都要有正反兩組測試（誤射消失 + 真陽性保留）。

## 2. advisory 與 fail-closed 是兩種機關，降噪策略相反

**事故**：worker 部署硬 gate 誤攔了「寫回歸測試」的 heredoc（測試斷言字串裡
含 'wrangler deploy'）。有人提議套用上面的引號/heredoc 遮罩來降噪。

**裁決過程（值得完整記住）**：v1 用「消費者黑名單」遮罩（heredoc 的消費者不是
shell 就遮罩）——被 review 用引號 argv 繞過 3 次打穿；v2 用「資料落地白名單」
——又被兩輪 held-out 測試各打穿一條（`<<-` tab 終止符、引號內 `<<` 幻影
heredoc）。最終結論：**regex/行解析器模擬 shell 解析超出安全可達範圍**，
該 gate 永久掃原文、寧可誤攔，17 條實證繞過樣本全部回歸測試釘死；誤攔的
正解放在 agent 側（deny 訊息引導改用檔案寫入工具，繞開 Bash 文本）。

**原則**：skill 預載這類 advisory 注入可以激進降噪（誤射只是噪音）；安全 gate
寧可誤攔不可漏攔（誤攔是摩擦，漏攔是事故），精準化要單獨開工單配完整
對抗測試，證明不了安全就不放鬆。

## 3. Gate 不能死鎖

**設計**：Stop gate 必須「攔一次即放行」——同一波編輯不二攔，且平台的
`stop_hook_active` 類重入標記永遠直通。否則 agent 卡在「結束→被攔→結束」循環。

**配套**：正因 Stop gate 軟，真正的硬 gate 放在 commit——「多按一次結束」
繞得過 Stop，繞不過 commit deny。強制力分層：軟 gate 保流暢，硬 gate 守關口。

**同族問題**：任何「收斂循環」機關（如對抗式 review loop）必須有硬預算上限。
真實事故：review skill 的唯一數值終止條件在某類 churn 下永不觸發，循環 200 輪。
修法 = 硬輪次預算 + carryover 煞車，且用 eval 守護「會終止」這個性質本身。

## 4. 「機關只靠 agent 在場」

**事故**：全套 hook gate 建成後全倉掃描發現——git hooks 全是 .sample、CI 只
build 不測試、deploy preflight 只查 git 衛生不跑測試。結論：所有機關只綁在
agent hook 層，人手 terminal commit、網頁 merge、任何非 agent 自動化全部繞過。

**修法優先序**：① CI 在 build 前加最便宜的硬檢查（lint/i18n/可信集），改動
最小擋面最大；② 給 server 類模組加離線可信集 CI；③ setup 腳本裝 git hooks
（core.hooksPath）作本地防線。三條路共用同一套可信集腳本，不另寫一套。

## 5. 閘後全軟

**事故**：自我評估發現硬 gate 全部集中在「進 git / 進 prod」閘口之前；閘口
之後——部署後驗證（真的去查了 /metrics 嗎）、review 證據、週報消費——全是
散文協議。且「warn 升 deny」寫了計畫但數據累積夠了也沒人升。

**教訓**：
- 生命週期畫一條線從 edit 到 prod 觀測，逐段標強制等級，「全軟段」就是下一批工單。
- warn 的升級條件要寫成可檢查的事實（「累積一週數據」），並把「到期未升級」
  本身作為 inbox 信號——否則過渡態變永久態。

## 6. 偵測器的視野窄於編輯面

**事故**：skill/規則的 .md 編輯不武裝驗證 ledger（偵測器只認 code 副檔名），
而 harness 可信集其實守著 .md 的不變量——結果帶著紅測試 commit 無人發現。

**原則**:「哪些檔案算 code」的定義必須跟著「可信集實際守什麼」走，不是跟著
直覺的副檔名清單走。新增一致性檢查時，同步檢查它守護的檔案類型是否在
編輯偵測範圍內。

## 7. 正面證據可被偽造

**事故**：review gate 檢查「review 通過」的方式一度是掃文本 marker——
而 marker 可以出現在 commit message 裡被無意（或有意）偽造。

**修法**：正面證據走獨立記帳通道（CLI 寫入事件帳本），gate 檢查帳本記錄的
存在性與時間窗，不信任何出現在自由文本裡的聲稱。

## 8. 收集端跑贏消費端

**現象**：反思 gate 上線後 inbox 條目高速累積（單日 +10），消費端一拍一條，
積壓迅速到 18 條。積壓本身不是失敗（收集端工作正常），但無觀測的積壓會
讓 inbox 退化成日記。

**修法**：積壓數進儀表板 + SessionStart 例行提醒（「行有餘力時消費一條」）；
積壓持續上升時，週 retro 要麼提高消費節奏、要麼把同族條目合併成一張工單
批量處理——兩者都是顯式決策，不是放著。

## 9. 帳本/清單雙寫必然漂移

**事故**：skip 清單同時存在於可信集腳本內和債務文檔裡，兩處各自演化。

**原則**：任何清單只允許一個 SSOT（ledger），所有消費者（腳本、儀表板、
文檔）從它讀。發現雙寫時的修法不是「同步一次」，是砍掉一邊。

## 10. 誤射降噪殺掉真陽性

**事故**：一次信噪修正差點把部署逃生偵測（`DEPLOY_FORCE=1` 路徑的記帳）
一起消滅——修的人只跑了「誤射消失」的測試。

**紀律**：降噪 PR 的測試清單必須成對：每個 suppressed case 配一個
still-fires case。review 時看到只有單邊測試直接退回。
