# Refine：粗想法 → 可開工任務書

模糊輸入直接開工的代價是方向錯了整輪重來——而且錯誤在交付時才暴露，
修正成本是開工前對齊的十倍以上。這個 skill 做的不是改寫用戶的話
（用戶的原話永遠保留），而是把它**展開**成一份雙方確認過的任務書；
確認後，任務書就是這輪工作的驗收依據。

這等於把「提示詞生成器」搬進對話流：用戶不需要學怎麼寫好提示，
只要說出想法，展開和補全由你完成，用戶只做確認和修正。

## 任務書模板（固定六段，總長控制在一屏內）

```
## 需求複述
<一兩句，用你的話說出你理解的目標——錯了用戶馬上能糾正>

## 意圖（why）
<這個需求背後想解決什麼問題、給誰用。從上下文推斷，推不出就列進開放問題>

## 驗收標準
<可驗證的形式：測試命令、可觀察的行為變化、產出物清單。禁止「做好做完」類定性詞>

## 本次不做
<明確劃出去的相鄰誘惑：順手重構、擴 scope、相關但獨立的問題>

## 開放問題（最多 3 個，沒有就刪掉這段）
<只列「答案會改變做法」的問題；能用合理默認補的不要問，寫進複述讓用戶看到即可>

## 執行建議
<自己做還是派工；派工的話給誰（模型層級）——層級判定照 model-tier-prompting>
```

## 流程

1. **展開**：按模板產出任務書。填不滿的段落用合理默認補上並標明是默認，
   不要空著等用戶餵——用戶改一份 80 分草稿比從零回答問卷快得多。
2. **確認**：有「答案會改變方向」的開放問題 → 停下等用戶答（互動場景可用
   選項式提問）；沒有 → 呈現任務書，等用戶說「可以」或改動。
3. **動工**：確認後照任務書執行，驗收對照任務書的驗收標準段逐條自查。
   任務書中途要改（發現新資訊），改動要讓用戶看到，不要靜默偏離。

## 邊界

- **不改寫用戶原話**：任務書是展開，不是替代。用戶說的話照原樣留在對話裡。
- **不對跟進型輸入使用**：「繼續」「commit」「好」「先這樣」這類不是需求。
- **任務書不是 PRD**：控制在一屏內。真需要完整技術設計的（跨系統、多階段），
  任務書的執行建議段直接指向工程自己的設計流程，不要在這裡寫長文。
- **已經清楚的需求不要硬展開**：用戶給了驗收標準和邊界的，直接開工，
  頂多一行複述確認。對清楚需求跑完整模板是浪費雙方時間。
