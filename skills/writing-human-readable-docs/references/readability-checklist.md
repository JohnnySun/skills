# 可读性 8 维度 checklist

按此逐项自检，每项 1-5 分，目标总分 ≥ 30/40。低于 4 分的维度需改。

## 1. 句子长度与复杂度

- 单句 ≤ 25 字（中文）。超长句拆成 2-3 句。
- 一句话不塞多个概念。嵌套解释（括号套括号、破折号套破折号）拆开。
- 检查方法：读出声，喘不上气 = 太长。

**反面**：「本次将这类编排写成一份 .workflow.js 脚本——脚本里定义了「分几步、每步起几个 AI 子任务、各步怎么串」。运行时由编排器按脚本执行：每一步的中间结果存在编排器的变量里，只有最终汇总答案回到 AI 对话中。」（一句话 3 个转折）
**正面**：拆成「写成 .workflow.js 脚本，定义分几步、每步起几个子任务、各步怎么串。运行时由编排器执行。中间结果存在编排器变量里，只有最终汇总回到对话。」

## 2. 段落密度

- 段落 ≤ 5 行（桌面端），移动端更短。
- 一段一要点。一段塞多个要点的，拆成列表或子段。
- 关键信息不埋在长段中部（扫读者只看段首段末）。

## 3. 信息层级

- 标题层级（H1→H2→H3）反映逻辑结构。
- 标题能让读者预判内容。泛标题（「新增能力」「使用方式」「关键设计选择」）加信息量或换具体标题。
- 同级标题结构一致（都用「问题→解法」或都用「此前/现」，不混）。

## 4. 扫读性

- 5 秒扫读能抓每节要点。
- 关键信息前置（front-loading）：段首给结论，后面展开。
- 善用加粗、列表、表格、代码块。
- 标题、加粗词作为扫读锚点。

## 5. 列表与表格的使用

- 并列信息（≥3 项）用列表，不用散文。
- 对比信息用表格（此前/现、旧/新、维度 A/B/C）。
- 步骤用有序列表，命令用代码块。

## 6. 术语密度与解释

- 首现术语有解释（括号或术语表）。
- 单位文字术语量不超载——一段抛 5+ 新术语的认知负荷太高（Miller 7±2）。
- 贯穿全文的术语集中到开头术语表，后续直接用。

## 7. 过渡与连贯

- 章节顺序符合读者预期：背景→问题→方案→使用→效果→行动→FAQ。
- 段间有逻辑过渡（「因此」「详见 §X」）。
- 同一概念全文叫法一致（不混用「harness / 工作台脚本 / 工作台自测」三种叫法）。
- 时态一致（不混「此前/现」和「之前/现在」）。

## 8. 冗余与重复

- 同一断言只详述一处，其他交叉引用。
- 同一解释不重复（如 submodule 在 3 处各解释一遍）。
- 多张内容重叠的表合并为一张。
- 废话凑数删掉。

## 评分基准

| 分数 | 含义 |
|------|------|
| 5 | 该维度做得很好，无需改 |
| 4 | 基本到位，有小优化空间 |
| 3 | 有明显问题，需改 |
| 2 | 问题严重，读者会卡住 |
| 1 | 该维度几乎没做到 |

总分 32-40 = 优秀，25-31 = 中等，<25 = 需重写。
