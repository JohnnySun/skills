# 理解設計（排版之上：決定「看完容易懂」的那一層）

排版決定「看上去清晰」，理解設計決定「看完真的懂了」。兩份文檔可以
排版同樣漂亮，一份讀完就能上手、另一份讀完還是一團霧——差異在這一層。

## 1. 按讀者任務組織，不按系統結構組織

工程師的本能是按系統結構寫（模組 A、模組 B、模組 C），但讀者帶著任務來
（「我想做 X」）。標題應該是讀者的問題，不是系統的零件名。

- 反面：「§3 閘道層 §4 訂單服務 §5 庫存模組」——讀者要自己拼裝任務路徑。
- 正面：「§3 如何建立訂單 §4 下單失敗怎麼排查」——系統結構藏在任務敘述裡。

**文檔類型不要混**（Diátaxis 框架）：教程（手把手第一次）、操作指南
（完成特定任務）、參考（查詢事實）、解釋（理解原理）是四種文檔，讀者
預期完全不同。一份文檔試圖同時當教程和參考手冊，是「又臭又長還難查」
的最常見根因——拆開，互相連結。

## 2. 心智模型先行

讀者理解新系統的方式是把它掛到已有模型上。先給一個**粗糙但方向正確**
的類比或一句話模型（「這本質上是一個帶審批的訊息佇列」），再展開細節
——細節有了掛點，記憶負荷減半。

紀律：類比必須聲明失效點（「與佇列不同的是，這裡的消費者可以拒收退回」）
——不聲明失效點的類比，會在讀者心裡種下錯誤推論。

## 3. 前向引用最小化

不要使用還沒解釋的概念（「詳見後文」是債）。概念的介紹順序 = 依賴順序：
被依賴的先講。寫完自查：從頭讀，每個術語在首次出現時是否已可理解
（術語表算「已解釋」）。長文檔各節盡量自包含——讀者是跳著讀的。

## 4. 負知識：預判並拆除錯誤模型

讀者不是空白的，是帶著「可能錯的舊模型」來的。好文檔會主動點名：
「你可能以為 X（因為舊版/常見工具是這樣），但這裡是 Y，因為 Z」。
一段負知識勝過三段正面描述——它精確命中讀者真實的困惑點。
收集來源：評審 agent 報告的誤解、真實用戶的提問、遷移前後的行為差異。

## 5. 具體性壓倒抽象性

- 數字給參照：「快 3 倍（30ms→10ms）」而不是「顯著提升」。
- 抽象規則配具體例子（先例後則，見 markdown-typography §4）。
- 「等等」「之類」是逃避：列得出就列全，列不全就給判準。

## 6. 標題可導航 + 檢索友好

- **目錄即摘要**：把所有標題連起來讀，應該像這份文檔的摘要——讀不通
  說明結構有問題。
- **為第二次閱讀優化**：讀者第二次來是「查」不是「讀」——關鍵詞要出現
  在標題和表格首列；FAQ 的問題用讀者的原話（帶錯誤術語也照用，那才
  搜得到），不用作者的規範術語。

## 7. 信任信號

過時的文檔比沒有文檔更糟（讀者按錯的做）。對外/長壽命文檔標注：
最後更新日期、適用版本範圍、負責人/反饋渠道。內容裡易漂移的細節
（數字、截圖、路徑）越少越長壽——這也是「數字用模糊量」原則的深層理由。

## 自查五問（寫完後過一遍，配合 9 維自檢）

1. 標題連讀是這份文檔的摘要嗎？
2. 每個術語首次出現時讀者已能理解嗎（無前向引用）？
3. 讀者最可能帶著哪個錯誤預期來？我拆除了嗎？
4. 這份文檔是教程/操作/參考/解釋中的哪一種？有沒有混型？
5. 第二次來查的人，能在 30 秒內定位到他要的段落嗎？
