# 圖表選型與設計（表格 / Mermaid / 何時該上圖）

文字、表格、圖各有擅長的資訊形態。好文檔的標誌之一：**在合適的位置用
合適的圖表**——讀者遇到流程時看到 flowchart、遇到多方互動時看到時序圖，
而不是三段「首先…然後…如果…否則…」的散文迷宮。圖表不是裝飾，是換一種
表達成本更低的載體。

## 1. 何時該上圖（觸發判據，寫大綱時逐節問）

| 正文出現的信號 | 該用的圖表 | Mermaid 語法 |
|---|---|---|
| 「首先→然後→如果→否則」≥3 個轉折 | 流程圖 | `flowchart TB/LR` |
| 多方來回（A 調 B、B 回 C、C 通知 A）≥3 次交互 | 時序圖 | `sequenceDiagram` |
| 「狀態從 X 變 Y」且狀態 ≥4 個 | 狀態機圖 | `stateDiagram-v2` |
| 描述系統的組成/層次/依賴關係 | 架構圖 | `flowchart` + subgraph |
| 里程碑、階段、先後排期 | 時間線 | `timeline` / `gantt` |
| 可枚舉的短事實對比（≥2 方 × ≥2 維度） | 表格 | Markdown 表格 |
| 概念的包含/分類關係 | 樹 / 巢狀列表 | 縮排列表即可 |

**反判據（不該上圖的情況）**：

- 要點只有兩三條——列表就夠，畫圖是過度包裝。
- 「偽圖」：把一段話塞進一個框、三個框排成一行卻沒有關係線——沒有
  結構資訊的圖是佔地方的裝飾。
- 裝飾性配圖：與理解無關的示意圖只會打斷閱讀節奏。
- 資訊還在頻繁變動的細節（圖比文字難改，易過時的內容先用列表）。

## 2. Mermaid 設計紀律

畫錯的圖比灰牆更勸退。逐條過：

1. **一圖一主題**。一張圖只回答一個問題（「請求怎麼流轉」或「狀態怎麼
   遷移」），不要合併。想講兩件事就畫兩張。
2. **節點數 ≤12**。超過就分層：先畫一張粗粒度總圖，細節拆成子圖放對應
   小節。20 節點的義大利麵圖等於沒畫。
3. **節點文字 ≤8 字**。節點裡放名詞短語，解釋放正文。長文字會把圖撐成
   表格。
4. **方向有語義**：步驟流程用 `TB`（上下，符合閱讀順序）；管道/時間/
   資料流用 `LR`（左右）。同文檔內同類圖方向一致。
5. **分支必帶標籤**。判斷節點的出邊寫「是/否」「成功/失敗」，不讓讀者
   猜哪條線是哪種情況。
6. **圖不孤立**（可及性鐵律）：圖前一句話預告「下圖是…」，圖後不重複
   細節；但**圖承載的關鍵結論正文必須有粗粒度版本**——Mermaid 渲染
   失敗（不支援的平台、郵件、PDF 轉換）時，讀者仍能從正文拿到主線。
7. **標籤語言與正文一致**（正文中文則節點中文；程式碼實體名保留原文）。
8. **避免交叉線**。線交叉 ≥3 處通常代表節點擺放或抽象層級有問題——
   重新分組，或升一層抽象。

## 2.5 Mermaid 與表格之外的視覺工具箱

Markdown 生態裡還有這些載體，各有不可替代的場景：

| 載體 | 不可替代的場景 | 紀律 |
|---|---|---|
| 目錄樹（`tree` 風格程式碼塊） | 展示檔案/模組結構 | 只列相關層級，`…` 省略無關項；每行可加行尾註解 |
| diff 塊（```diff） | 改動前後對照 | 只放關鍵行 ±2 行上下文，不貼全檔 |
| 輸入/輸出樣例對 | API、命令、格式說明 | 真實可跑的樣例；輸入輸出緊鄰放 |
| Callout（`> [!NOTE]` / `[!WARNING]`） | 警告、前提、旁註 | 全文 ≤5 個；WARNING 只給真的會出事的 |
| 任務清單（`- [ ]`） | 讀者要逐項完成/自查的事 | 每項可獨立勾掉；不當普通列表用 |
| ASCII 框圖 | mermaid 不渲染的環境（純終端、註釋內） | 寧簡勿繁；複雜了就回 mermaid + 正文粗版 |
| 截圖＋標註 | UI 操作、視覺結果 | 圈紅/箭頭指目標；配 alt 文字；截最小區域 |
| 狀態 emoji（✅❌⚠️） | 表格中的三態標記 | 首次給圖例；全文 ≤3 種，不當裝飾撒 |
| Mermaid 冷門型（pie/mindmap/erDiagram/journey/gantt） | 佔比、腦圖、資料模型、旅程、排期 | 先問「表格是不是更清楚」——冷門型渲染支援差，慎用 |

選型總原則：**先問資訊形態，再挑最樸素的夠用載體**——表格能說清就不
畫圖，列表能說清就不建表。載體越花哨，維護成本與渲染風險越高。

## 3. 表格設計（比「對比用表格」再深一層）

1. **首列是查找鍵**。讀者拿什麼找行（場景名、命令、角色），什麼就放
   第一列；其餘列按「最常查 → 最少查」從左到右排。
2. **行序有意義**。按重要性、時間、頻率或字母排——不要按想到的順序。
3. **單元格一行內**。放判斷、數值、短語；需要解釋的內容放表後的段落
   或腳註，不要在格子裡寫作文。
4. **相同的東西不進表**。對比表只列有差異的維度；全部一樣的行刪掉，
   在表前一句話帶過（「兩方案的部署方式相同，差異如下」）。
5. **每表有表意**。表前一句話說「這張表回答什麼問題」；沒有引導句的表
   讀者要自己歸納表意，成本翻倍。
6. **符號表要有圖例**。用 ✓/✗/－ 時，首次出現時說明含義（尤其「－」是
   「不適用」還是「未測」）。
7. **寬表拆分或轉置**：>5 列先問「讀者是按行查還是按列查」——按行查就
   轉置；維度太多就按主題拆成兩張表。
8. **對齊**：數字右對齊（比大小），文字左對齊。

## 4. 位置與節奏

- **同屏原則**：圖表緊貼它解釋的正文（之後，不是之前隔三段）；讀者
  視線不應該在圖與相關文字之間翻頁。
- **複雜圖先預告**：一句話說「下圖展示 X，重點看 Y」再上圖，給讀者
  掃描的目標；簡單表格可直接上。
- **密度平衡**：連續兩屏全是圖表與連續兩屏全是散文同樣單調（見
  markdown-typography.md §3 屏分佈規劃）——圖表也是視覺錨點的一種，
  計入每屏錨點配比。
- **附錄承接溢出**：正文只放主線必需的圖；輔助性大圖（完整狀態機、
  全量架構）放附錄並在正文連結。

## 5. Before / After 小樣

**Before（散文迷宮）**：

> 使用者提交表單後，系統先做本地校驗，校驗不過就地提示；通過後發送到
> 閘道，閘道檢查登入態，未登入跳轉登入頁，已登入轉發給訂單服務，訂單
> 服務檢查庫存，庫存不足回錯誤碼 409 由前端提示補貨時間，庫存充足則
> 建立訂單並回訂單號，前端拿到訂單號後跳轉支付頁。

**After**：

> 提交後的處理流程如下（兩個攔截點：登入態、庫存）：
>
> ```mermaid
> flowchart TB
>     A[提交表單] --> B{本地校驗}
>     B -- 不過 --> B1[就地提示]
>     B -- 通過 --> C{閘道：已登入?}
>     C -- 否 --> C1[跳轉登入頁]
>     C -- 是 --> D{訂單服務：庫存?}
>     D -- 不足 --> D1[409：提示補貨時間]
>     D -- 充足 --> E[建立訂單 → 跳轉支付頁]
> ```
>
> 主線一句話：校驗與登入都通過且有庫存時，建單並進入支付。

同樣的資訊：前者讀者要在腦中自己建圖；後者圖給結構、正文給主線，
渲染失敗時最後一句話仍保住可讀性。
