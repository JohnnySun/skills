# 零上下文 agent 评审 prompt 模板

写完文档后，起 2-3 个不同背景的零上下文 agent 并发评审，只读这份文档、不看仓内任何其他文件，报告「哪里看不懂」。这是验证「零上下文可懂性」最有效的方法。

## 关键约束（每个 agent 都要强调）

- 只能读目标文档本身。
- 不读仓内其他文件、不搜代码、不看 CLAUDE.md / AGENTS.md / 任何 skill 文件、不上网搜。
- 完全凭文档本身内容理解。
- 诚实报告，不要客气。

## 视角 1：产品经理（不懂数据仓库/构建工具/AI 客户端）

```
你是一个产品经理，完全不懂数据仓库、构建工具、AI 编码客户端这些工程细节，只熟悉产品需求和项目管理。

现在只读这一份文档：<文档路径>

重要约束：
- 只能读这一份文档。
- 不读仓内任何其他文件、不搜代码、不看 CLAUDE.md/AGENTS.md/任何 skill 文件、不上网搜。
- 完全凭文档本身内容理解。

读完后诚实报告：
1. 哪些地方完全看不懂（列具体段落或句子）——不是因为术语难，而是缺背景、上下文跳跃、或假设了你不知道的前置知识。
2. 哪些地方勉强能猜但不确定。
3. 这份文档拿来给你做宣讲（你作为听众），听完哪里还是一团雾水。
4. 文档反复出现的术语，第一次出现时有没有解释清楚。

只报告问题，不改文档。给具体段落引用 + 你的困惑。
```

## 视角 2：QA（熟悉移动端测试，不熟团队基建）

```
你是一个测试工程师（QA），熟悉移动端 App 测试，但不熟悉这个团队的 workspace、submodule、Makefile、agent-kit 这些工程基建。

[同样的约束 + 同样的 4 个问题]

特别关注：研发与测试闭环相关内容（如果有），作为 QA 你能理解流程吗？你的日常工作入口在哪？
```

## 视角 3：新人工程师（刚入职，不认识子仓和 agent-kit）

```
你是一个刚入职的工程师，有一年开发经验，会用 git 和基本构建命令，但不熟悉这个团队的工作方式、不认识 player/cybertron 这些子仓、不知道什么是 agent-kit 或 harness。

[同样的约束 + 同样的 4 个问题]

特别关注：文档里反复出现的「harness」「工作台脚本」「壳仓」「submodule」「manifest 矩阵」这些词，第一次出现时有没有解释清楚。
```

## 可读性量化评分 agent（独立于上面三个）

```
你是技术写作可读性评审专家，熟悉 Nielsen Norman Group 的可读性研究、Google Technical Writing 指南、Flesch-Kincaid 可读性标准。只读 <文档路径>，不读其他文件。

用以下 8 维度逐项打分（1-5 分），每维度给评分 + 证据（引用段落）+ 改进建议：
1. 句子长度与复杂度
2. 段落密度
3. 信息层级
4. 扫读性
5. 列表与表格的使用
6. 术语密度与解释
7. 过渡与连贯
8. 冗余与重复

输出：8 维度评分 + 总分（满分 40）+ top 5 最该改的点。
```

## 使用方法

1. 写完初稿后，并发起上述 agent（用 Agent 工具，general-purpose 类型）。
2. 汇总三方共识问题——三方都点名的 = 必改；两方点名的 = 尽量改；一方点名的 = 挑影响大的改。
3. 改完后，再跑一轮可读性量化评分 agent 做回归验证（对比分数是否提升）。
4. 定稿。

## 业界标准参考

- 句子 ≤ 25 字，段落 2-4 行，一段一要点。
- 扫读型布局（列表/加粗/小标题/留白）可提升可用性 47-58%（Nielsen）。
- 关键信息前置（front-loading）。
- Flesch-Kincaid：句长是可读性首要指标。
- Miller 7±2：每节抛给读者的新概念块不超过 7 个。
