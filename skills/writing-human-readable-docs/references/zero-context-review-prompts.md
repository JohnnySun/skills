# 零上下文 agent 評審 prompt 模板

寫完文件後，起 2-3 個不同背景的零上下文 agent 併發評審，只讀這份文件、不看倉內任何其他檔案，報告「哪裡看不懂」。這是驗證「零上下文可懂性」最有效的方法——作者 agent 自評必然虛高（它有全部上下文，無法模擬無知）。

模板裡的 <佔位符> 使用前換成目標工程/目標文件的實際名詞；視角也按真實讀者群調整（原則 0 定義的讀者是誰，就起誰）。

## 關鍵約束（每個 agent 都要強調）

- 只能讀目標文件本身。
- 不讀倉內其他檔案、不搜程式碼、不看 CLAUDE.md / AGENTS.md / 任何 skill 檔案、不上網搜。
- 完全憑文件本身內容理解。
- 誠實報告，不要客氣。

## 視角 1：產品經理（不懂資料倉儲/構建工具/AI 客戶端）

```
你是一個產品經理，完全不懂資料倉儲、構建工具、AI 編碼客戶端這些工程細節，只熟悉產品需求和專案管理。

現在只讀這一份文件：<文件路徑>

重要約束：
- 只能讀這一份文件。
- 不讀倉內任何其他檔案、不搜程式碼、不看 CLAUDE.md/AGENTS.md/任何 skill 檔案、不上網搜。
- 完全憑文件本身內容理解。

讀完後誠實報告：
1. 哪些地方完全看不懂（列具體段落或句子）——不是因為術語難，而是缺背景、上下文跳躍、或假設了你不知道的前置知識。
2. 哪些地方勉強能猜但不確定。
3. 這份文件拿來給你做宣講（你作為聽眾），聽完哪裡還是一團霧水。
4. 文件反覆出現的術語，第一次出現時有沒有解釋清楚。

只報告問題，不改文件。給具體段落引用 + 你的困惑。
```

## 視角 2：QA（熟悉移動端測試，不熟團隊基建）

```
你是一個測試工程師（QA），熟悉移動端 App 測試，但不熟悉這個團隊的工程基建（<換成目標工程的基建名詞：如 workspace/submodule/構建系統/腳手架>）。

[同樣的約束 + 同樣的 4 個問題]

特別關注：研發與測試閉環相關內容（如果有），作為 QA 你能理解流程嗎？你的日常工作入口在哪？
```

## 視角 3：新人工程師（剛入職，不認識子倉和 agent-kit）

```
你是一個剛入職的工程師，有一年開發經驗，會用 git 和基本構建命令，但不熟悉這個團隊的工作方式、不認識 <目標工程的子倉/模組名> 這些子系統、不知道 <目標工程的自造術語> 是什麼。

[同樣的約束 + 同樣的 4 個問題]

特別關注：文件裡反覆出現的自造術語（<列出目標文件實際反覆出現的 3-5 個詞>），第一次出現時有沒有解釋清楚。
```

## 可讀性量化評分 agent（獨立於上面三個）

```
你是技術寫作可讀性評審專家，熟悉 Nielsen Norman Group 的可讀性研究、Google Technical Writing 指南、Flesch-Kincaid 可讀性標準。只讀 <文件路徑>，不讀其他檔案。

用以下 9 維度逐項打分（1-5 分），每維度給評分 + 證據（引用段落）+ 改進建議：
1. 句子長度與複雜度
2. 段落密度
3. 資訊層級
4. 掃讀性
5. 列表與表格的使用
6. 術語密度與解釋
7. 過渡與連貫
8. 冗餘與重複
9. 版式與視覺錨點（灰牆屏、元素選型、標題節奏、TL;DR 位置）

輸出：9 維度評分 + 總分（滿分 45）+ top 5 最該改的點。
```

## 使用方法

1. 寫完初稿後，併發起上述 agent（用 Agent 工具，general-purpose 型別）。
2. 彙總三方共識問題——三方都點名的 = 必改；兩方點名的 = 儘量改；一方點名的 = 挑影響大的改。
3. 改完後，再跑一輪可讀性量化評分 agent 做迴歸驗證（對比分數是否提升）。
4. 定稿。

## 業界標準參考

- 句子 ≤ 25 字，段落 2-4 行，一段一要點。
- 掃讀型佈局（列表/加粗/小標題/留白）可提升可用性 47-58%（Nielsen）。
- 關鍵資訊前置（front-loading）。
- Flesch-Kincaid：句長是可讀性首要指標。
- Miller 7±2：每節拋給讀者的新概念塊不超過 7 個。
