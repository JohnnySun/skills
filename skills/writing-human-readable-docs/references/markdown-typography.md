# Markdown 排版與版式規範（灰牆防治）

排版是設計，不是排水。一坨灰壓壓的文字（gray wall）會讓讀者在開啟的
3 秒內決定「回頭再看」——然後永遠不看。本篇管三件事：文字密度、
Markdown 元素選型、文件級版式規劃。

## 1. 文字密度（灰牆防治三條硬規則）

1. **每屏至少一個視覺錨點**。一屏約 20-25 行；連續 8 行以上純散文即灰牆。
   錨點 = 標題 / 列表 / 表格 / 程式碼塊 / 加粗關鍵詞 / 引用塊。寫完滾一遍
   全文，哪一屏沒有錨點就重排哪一屏。
2. **段落 2-4 行，段間必空行**。一段一要點；要點講完就換段，不要因為
   「話題相近」把兩個要點擠在一段。
3. **加粗是稀缺資源**。每段最多 1-2 處，加粗的是「掃讀者必須捕獲的詞」，
   不是「作者覺得重要的整句」。全加粗等於沒加粗。中文避免斜體（渲染
   效果差），強調一律加粗。

## 2. Markdown 元素選型表（資訊形態 → 元素）

| 資訊形態 | 用什麼 | 不要用什麼 |
|---|---|---|
| 並列項 ≥3 | 無序列表 | 頓號串在散文裡 |
| 有先後的步驟 | 有序列表（一步一行為） | 「先…然後…接著…最後…」長句 |
| 兩方以上對比（舊/新、方案 A/B） | 表格 | 交替段落 |
| 命令、路徑、程式碼、欄位名 | 反引號 / 程式碼塊 | 引號或裸文字 |
| 旁註、警告、前提 | 引用塊（> ）或 callout | 括號塞進正文 |
| 主題切換 | 新的 H2/H3 | 水平線濫用（hr 全文 ≤2 條） |

配套紀律：

- **列表巢狀 ≤2 層**。第三層出現時，把外層拆成小標題。
- **列表項互相對稱**：都是名詞短語、或都是「動詞開頭的指令」，不混。
  單項 ≤2 行，超了說明它其實是一個段落。
- **表格 ≤5 列，單元格一行內**。塞不下的內容回到散文或列表——表格是
  給「可列舉的短事實」的，不是給解釋的。
- **標題節奏**：每 3-8 段一個小標題；層級不跳（H2 下不直接 H4）；
  標題本身帶資訊量（「為什麼遷移」而不是「背景」，能預判內容）。

## 3. 文件級版式規劃（寫大綱時一起畫）

### TL;DR 規範

第一屏、術語表之前，3-5 句或 3-5 個 bullet，回答：這是什麼 / 跟讀者
什麼關係 / 讀者要做什麼 / 什麼時候生效。對外發布與宣講類文件必有；
標題就叫 `TL;DR` 或「一分鐘版本」，讓掃讀者知道讀完這一塊就可以走。

### 屏分佈規劃

列大綱時不只列「節」，還要預估「屏」：每一節大概幾屏、每屏讀者看到
什麼錨點。經驗配比——一屏內：標題 1 個 + 散文 ≤8 行 + 一個結構化元素
（列表/表格/程式碼塊）。連續兩屏同一種形態（全是表格、全是列表）也是
單調，穿插散文過渡。

### 換行與行寬規律

- **原始碼級**：hard wrap（原始碼約 80-100 字元斷行）還是軟換行，跟目標
  倉庫既有慣例走，全文統一，不混用。
- **渲染級**：作者管不了渲染寬度，但管得了句長與段長——這正是句子
  ≤25 字、段落 ≤4 行兩條規則的版式意義：任何渲染寬度下都不結成灰牆。
- 中英文混排：中英文之間留空格；全文標點全半形一致。

## 4. 漸進式講解（結構層面的易讀性）

- **金字塔**：全文層面結論先行（TL;DR），各節層面段首句即該節結論，
  細節往後排。讀者在任何深度停下來，拿到的都是完整的粗粒度版本。
- **洋蔥**：複雜機制每節先給「一句話版本」，再給展開版。兩版之間用
  空行或小標題隔開，讓不需要細節的讀者明確知道可以跳。
- **先具象後抽象**：先給一個讀者熟悉場景的例子，再給一般化規則。
  例子扛認知負荷的能力遠大於定義。
- **新概念節流**：一屏內新概念 ≤3 個（比 Miller 7±2 更嚴——讀者是
  零上下文）。超了就拆節，或把次要概念挪進術語表/腳註。

## 5. Before / After 小樣

**Before（灰牆）**：

> 本次升級將構建流程從原有的 Jenkins 流水線遷移到 GitHub Actions，主要原因是 Jenkins 維護成本高且構建機資源緊張經常排隊，同時舊流水線的配置分散在多個 Jenkinsfile 中難以維護，新的流水線統一使用倉庫根目錄的 workflows 目錄管理並且支援按需觸發和矩陣構建，各團隊需要在下週一之前把自己專案的構建配置遷移過來，具體做法是複製模板檔案並修改其中的專案名稱和構建命令，遷移完成後在舊流水線頁面點擊停用……

**After**：

> **TL;DR**：CI 從 Jenkins 遷到 GitHub Actions，**下週一生效**。你需要
> 做一件事：按下面三步把自己專案的構建配置遷過來（約 15 分鐘）。
>
> 為什麼遷：
> - Jenkins 構建機排隊嚴重，維護成本高
> - 舊配置分散在多個 Jenkinsfile，難以維護
>
> 遷移三步：
> 1. 複製模板：`cp .github/workflows/template.yml .github/workflows/<專案名>.yml`
> 2. 改兩處：專案名稱、構建命令
> 3. 在舊流水線頁面點「停用」

同樣的資訊量：前者一屏灰牆零錨點，後者 TL;DR + 列表 + 程式碼塊 + 加粗
截止時間，5 秒抓到全部要點。
