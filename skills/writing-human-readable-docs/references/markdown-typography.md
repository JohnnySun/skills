# Markdown 排版与版式规范（灰墙防治）

排版是设计，不是排水。一坨灰压压的文字（gray wall）会让读者在打开的
3 秒内决定「回头再看」——然后永远不看。本篇管三件事：文字密度、
Markdown 元素选型、文档级版式规划。

## 1. 文字密度（灰墙防治三条硬规则）

1. **每屏至少一个视觉锚点**。一屏约 20-25 行；连续 8 行以上纯散文即灰墙。
   锚点 = 标题 / 列表 / 表格 / 代码块 / 加粗关键词 / 引用块。写完滚一遍
   全文，哪一屏没有锚点就重排哪一屏。
2. **段落 2-4 行，段间必空行**。一段一要点；要点讲完就换段，不要因为
   「话题相近」把两个要点挤在一段。
3. **加粗是稀缺资源**。每段最多 1-2 处，加粗的是「扫读者必须捕获的词」，
   不是「作者觉得重要的整句」。全加粗等于没加粗。中文避免斜体（渲染
   效果差），强调一律加粗。

## 2. Markdown 元素选型表（信息形态 → 元素）

| 信息形态 | 用什么 | 不要用什么 |
|---|---|---|
| 并列项 ≥3 | 无序列表 | 顿号串在散文里 |
| 有先后的步骤 | 有序列表（一步一行为） | 「先…然后…接着…最后…」长句 |
| 两方以上对比（旧/新、方案 A/B） | 表格 | 交替段落 |
| 命令、路径、代码、字段名 | 反引号 / 代码块 | 引号或裸文字 |
| 旁注、警告、前提 | 引用块（> ）或 callout | 括号塞进正文 |
| 主题切换 | 新的 H2/H3 | 水平线滥用（hr 全文 ≤2 条） |

配套纪律：

- **列表嵌套 ≤2 层**。第三层出现时，把外层拆成小标题。
- **列表项互相对称**：都是名词短语、或都是「动词开头的指令」，不混。
  单项 ≤2 行，超了说明它其实是一个段落。
- **表格 ≤5 列，单元格一行内**。塞不下的内容回到散文或列表——表格是
  给「可枚举的短事实」的，不是给解释的。
- **标题节奏**：每 3-8 段一个小标题；层级不跳（H2 下不直接 H4）；
  标题本身带信息量（「为什么迁移」而不是「背景」，能预判内容）。

## 3. 文档级版式规划（写大纲时一起画）

### TL;DR 规范

第一屏、术语表之前，3-5 句或 3-5 个 bullet，回答：这是什么 / 跟读者
什么关系 / 读者要做什么 / 什么时候生效。对外发布与宣讲类文档必有；
标题就叫 `TL;DR` 或「一分钟版本」，让扫读者知道读完这一块就可以走。

### 屏分布规划

列大纲时不只列「节」，还要预估「屏」：每一节大概几屏、每屏读者看到
什么锚点。经验配比——一屏内：标题 1 个 + 散文 ≤8 行 + 一个结构化元素
（列表/表格/代码块）。连续两屏同一种形态（全是表格、全是列表）也是
单调，穿插散文过渡。

### 换行与行宽规律

- **源码级**：hard wrap（源码约 80-100 字符断行）还是软换行，跟目标
  仓库既有惯例走，全文统一，不混用。
- **渲染级**：作者管不了渲染宽度，但管得了句长与段长——这正是句子
  ≤25 字、段落 ≤4 行两条规则的版式意义：任何渲染宽度下都不结成灰墙。
- 中英文混排：中英文之间留空格；全文标点全半角一致。

## 4. 渐进式讲解（结构层面的易读性）

- **金字塔**：全文层面结论先行（TL;DR），各节层面段首句即该节结论，
  细节往后排。读者在任何深度停下来，拿到的都是完整的粗粒度版本。
- **洋葱**：复杂机制每节先给「一句话版本」，再给展开版。两版之间用
  空行或小标题隔开，让不需要细节的读者明确知道可以跳。
- **先具象后抽象**：先给一个读者熟悉场景的例子，再给一般化规则。
  例子扛认知负荷的能力远大于定义。
- **新概念节流**：一屏内新概念 ≤3 个（比 Miller 7±2 更严——读者是
  零上下文）。超了就拆节，或把次要概念挪进术语表/脚注。

## 5. Before / After 小样

**Before（灰墙）**：

> 本次升级将构建流程从原有的 Jenkins 流水线迁移到 GitHub Actions，主要原因是 Jenkins 维护成本高且构建机资源紧张经常排队，同时旧流水线的配置分散在多个 Jenkinsfile 中难以维护，新的流水线统一使用仓库根目录的 workflows 目录管理并且支持按需触发和矩阵构建，各团队需要在下周一之前把自己项目的构建配置迁移过来，具体做法是复制模板文件并修改其中的项目名称和构建命令，迁移完成后在旧流水线页面点击停用……

**After**：

> **TL;DR**：CI 从 Jenkins 迁到 GitHub Actions，**下周一生效**。你需要
> 做一件事：按下面三步把自己项目的构建配置迁过来（约 15 分钟）。
>
> 为什么迁：
> - Jenkins 构建机排队严重，维护成本高
> - 旧配置分散在多个 Jenkinsfile，难以维护
>
> 迁移三步：
> 1. 复制模板：`cp .github/workflows/template.yml .github/workflows/<项目名>.yml`
> 2. 改两处：项目名称、构建命令
> 3. 在旧流水线页面点「停用」

同样的信息量：前者一屏灰墙零锚点，后者 TL;DR + 列表 + 代码块 + 加粗
截止时间，5 秒抓到全部要点。
