# RelayGW v2 灰度切流與 Affinity 遷移：你的服務要做什麼

## TL;DR

RelayGW（內部請求轉發閘道）正在把 v1 逐步換成 v2,採**灰度切流**：先讓
一部分服務走新版本,確認沒問題再擴大範圍。目前已經有 5% 的 tenant（以
服務／團隊為單位分組）在走 v2,比例會在**下週三擴大到 50%**、**下下週一
全量**。如果你的服務在灰度名單內,需要確認以下幾件事：**Redis key 格式**、
**憑證**、**逾時設定**、**監控指標**、**Redis 容量**。詳細清單見下方
「你要做的事」。

## 術語表

| 術語 | 白話解釋 |
|---|---|
| RelayGW（v1 / v2） | 內部的請求轉發閘道服務,呼叫下游前會先經過它;v1、v2 是這次要遷移的兩個版本 |
| 灰度切流 | 新版本不是一次全部上線,而是先讓一小部分流量（這裡按 tenant 分組）走新版本,確認穩定後再逐步擴大比例 |
| Affinity（親和性）／ affinity key | 一筆記錄同一個呼叫方後續請求要固定路由到哪個上游連線的 Redis 資料,本次遷移改的就是這筆資料的存放格式 |
| Bucket 分片 | 把同一組 affinity 資料拆成多個桶,降低多個請求同時搶同一把 key 的機率 |
| Singleflight 擊穿 | 一種「同一時間只讓一個請求真正去建立連線,其他請求等結果」的去重機制;在高並發下這個機制會失效（業界稱「擊穿」）,導致多個請求同時搶著建立上游連線 |
| TTL | Redis 資料的存活時間（秒）,到期會自動失效 |
| mTLS | 雙向 TLS：不只你驗 server 的憑證,server 也會反過來驗你的憑證 |
| Vault | 公司的密鑰／憑證管理系統 |
| 雙寫 | 遷移過渡期間,同一份 affinity 資料會同時寫入 v1 和 v2 兩種格式,確保切換過程中兩邊都讀得到 |

## 背景：為什麼要做這次遷移

v1 的 singleflight 去重機制在高並發下會擊穿,導致多個請求同時去建立
上游連線,拖慢回應也加重上游負擔。v2 引入 bucket 分片降低撞鍵機率,
這是本次升級的核心動機。完整的壓測數據記在內部設計文件 **RFC-2841**。

## 這次變更會如何影響你的服務

如果你的服務**在灰度名單內**,以下 6 件事會影響你：

- **Affinity key 格式變了**。如果你的服務有直接讀 Redis 裡的 affinity key,
  格式從 `rg:aff:{cid}` 換成 `rg2:aff:{cid}:{bucket}`,TTL 也從 300 秒
  延長到 900 秒。
- **舊的 prewarm 腳本失效**。`scripts/prewarm.sh` 是照舊 key 格式寫的,
  對 v2 無效,遷移期間請先停用。（新腳本的路徑原文未提供,需要的話請洽
  `@relay-oncall`。）
- **連線方式改成 mTLS**。v2 走 `relay-v2.internal:8443`,要先去 Vault
  的 `secret/relay/v2/client` 拉憑證;v1 的 Basic Auth 在 v2 不通,
  遇到 401 請先檢查憑證是否正確拉取。
- **預設逾時收緊**。connect timeout 從 5s 降到 2s、read timeout 從 60s
  降到 30s,原因是實測 p99 回應只要 1.2s,逾時設太長反而讓故障時的排隊
  更嚴重。如果你的呼叫真的需要更長逾時（例如批次介面）,可以在 header 帶
  `X-Relay-Timeout` 覆蓋,上限 120s,超過會被系統強制降到上限。
- **監控指標改名**。`relay_request_total` 這類指標前綴改成
  `rgw2_request_total`,既有的 dashboard 需要自己更新;Grafana 上有
  `RGW2-template` 可以直接 clone。
- **Redis 記憶體會漲**。灰度（雙寫）期間 v1、v2 的 key 會同時寫入,
  Redis 記憶體預估上漲約 15%,容量比較吃緊的叢集要自行評估;灰度結束後
  v1 的 key 會自然過期,不需要手動清理。

## 灰度排程

| 階段 | 流量比例 | 時間 |
|---|---|---|
| 目前 | 5% | 已開始 |
| 下一階段 | 50% | 下週三 |
| 全量 | 100% | 下下週一 |

> 上面的「下週三／下下週一」是本文件發出時的相對時間。如果你是晚一段
> 時間才看到這份文件,實際切流進度請跟 `@relay-oncall` 確認。

## 你要做的事（Checklist）

- [ ] 服務有直接讀 Redis 的 affinity key → 改成新格式 `rg2:aff:{cid}:{bucket}`
- [ ] 停用舊的 `scripts/prewarm.sh`
- [ ] 服務在灰度名單內 → 去 Vault `secret/relay/v2/client` 拉 mTLS 憑證
- [ ] 更新自己的監控 dashboard（clone `RGW2-template`）
- [ ] 批次介面需要長逾時 → 加 header `X-Relay-Timeout`（上限 120s）
- [ ] 評估自己叢集的 Redis 容量（雙寫期間約 +15%）

## FAQ / 已知取捨

- **遇到 401 怎麼辦？** 先檢查 mTLS 憑證是否正確拉取,v2 不接受 v1 的
  Basic Auth。
- **想了解 singleflight 擊穿的壓測細節？** 見內部設計文件 RFC-2841。
- **新的 prewarm 腳本在哪？** 原公告沒有給出路徑,目前只能請洽
  `@relay-oncall` 確認。
- **有問題找誰？** 透過 `@relay-oncall` 詢問,不要私訊個人。
