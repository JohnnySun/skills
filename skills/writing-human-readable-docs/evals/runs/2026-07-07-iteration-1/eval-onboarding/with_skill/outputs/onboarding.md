# 第一次把服務部署到 staging

這份文件教你完成入職後第一次部署：把一個服務發布到 staging 環境。跟你的關係是——這是你接下來每次上 staging 都會重複的動作，現在學會一次，之後就是肌肉記憶。讀完後你要能做到：申請好權限、跑完部署指令、確認服務真的正常，並且知道出錯了怎麼退回。整個流程（不含權限審批等待）大約 5-10 分鐘。

## 術語表

| 術語 | 意思 |
|---|---|
| staging | 正式上線前的測試環境，跟正式環境（production）分開，用來在真實用戶看到之前先驗證 |
| `dctl` | 部署用的命令列工具，本文所有指令都靠它 |
| deployer 角色 | 一種權限身份，沒有這個角色就無法對 staging 執行部署指令 |
| VPN | 連進公司內網的通道，`dctl` 的指令需要透過它才能連到 staging 環境 |

## 開始之前：兩個前置條件

**這兩件事都要先完成，缺一個都會卡住：**

1. **申請 deployer 角色**——到 IT portal 提出申請。審批大約需要**半天**，所以建議今天就申請，不要等到要部署的當下才申請。
2. **連上公司 VPN**——每次要跑部署指令之前都要先確認 VPN 是連著的（細節見下方「常見坑」）。

## 部署步驟

> 以下 `<service>` 換成你要部署的服務名稱。

**第 1 步：登入 staging**

```
dctl login staging
```

你應該看到登入成功的提示。如果指令卡住不動、遲遲沒反應，先看本文最後的「常見坑」。

**第 2 步：執行部署**

```
dctl deploy <service> --env staging
```

這一步大約需要 **3 分鐘**。過程中會有進度輸出，等它跑完再進行下一步。

**第 3 步：驗證部署成功**

做兩件事，兩件都要過：

- 執行 `dctl status <service>`，畫面應該顯示 **healthy**。
- 用瀏覽器打開 `https://staging.example.com/healthz`，頁面（或回應）應該是 **200**。

只要有一項不是預期結果，就先不要當作部署完成，回頭檢查上一步是否有錯誤訊息。

## 部署錯了怎麼辦：回滾

如果部署後發現不對勁，執行：

```
dctl rollback <service>
```

這會把服務退回部署前的狀態。

## 常見坑

**`dctl login` 卡住不動、一直等到逾時，但畫面不會提示「VPN 沒連上」。**

這是目前最容易踩的坑：問題根源通常是 VPN 沒連上，但工具本身不會明講。如果 `dctl login` 卡超過幾秒鐘沒反應，第一件事就是回頭確認 VPN 連線狀態，而不是繼續等或重跑指令。
