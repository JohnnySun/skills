# Writing Human-Readable Docs（寫人類可讀的技術文件）

## 目標

寫出「零上下文聽眾也能看懂」的技術文件——宣講稿、升級說明、公告、上手指南。核心判據：一個剛加入的新人或跨職能同事（產品/QA），不讀倉內任何其他檔案，只憑這一份文件，能不能看懂。

## 何時觸發

- 使用者說「寫個宣講文件 / 升級說明 / briefing / 公告」
- 使用者反饋「文件像天書 / 看不懂 / 語氣怪 / 太複雜」
- 使用者要 review 一份已有文件的可讀性
- 寫給跨職能聽眾（不只工程師）的任何技術說明

## 核心原則（按重要性排序）

### 0. 先定義讀者與成功動作（寫任何字之前）

一句話寫下：**讀者是誰、他們已經知道什麼、讀完這份文件他們要能做什麼**。
文件的驗收標準不是「讀者看懂了」，而是「讀者能完成那個動作」（升級、遷移、
配置、決策、轉述給別人）。這句話決定後面所有取捨：解釋多深、砍掉哪些
工程細節、行動指令放多前。沒有這句話的文件，最常見的死法是「工程師寫給
自己看的備忘錄」。

### 1. 零上下文假設（最關鍵）

假設讀者不知道任何背景：不認識倉裡的子倉、不懂 harness / agent-kit 是什麼、不知道團隊在用 AI 客戶端寫程式碼。所有前提先鋪墊，再用。

反面：直接拋「274 個測試針對 harness 本身」——讀者不知道 harness 是什麼、274 是什麼。
正面：先說「團隊用 AI 客戶端寫程式碼，工作臺是團隊的工作區，harness 指工作臺外層的腳手架腳本」，再說 274。

### 2. 術語表前置

貫穿全文的術語，在文件開頭集中定義一次（一個表格），後續直接用，不重複解釋。

反面：submodule 在 3 處各解釋一遍「git 的子倉庫功能…」。
正面：開頭術語表定義一次，正文只用「submodule」。

### 3. 去重——同一資訊只詳述一處

同一斷言（如「測試邊界」「簽名兩條路徑」）只在最合適的一節詳述，其他處用「見 §X」交叉引用。

反面：簽名兩條路徑在 §2.4、§3、§5.6、§6.1、§10 FAQ 五處各寫一遍。
正面：§6.1（使用處）詳述，其他處「見 §6.1」。

### 4. 結構化優於散文

並列資訊用列表或表格；「此前 / 現」對比用加粗分點；案例用對稱結構（場景/問題/正解）；變化總覽用對照表。

反面：一段 150 字散文塞「場景 + 舊做法 + 問題 + 新方案 + 原因」。
正面：拆成「- 場景 / - 問題 / - 正解」三行。

### 5. 版式是設計，不是排水（灰牆防治）

一坨灰壓壓的連續散文會讓讀者 3 秒內放棄。三條硬規則：**每屏至少一個
視覺錨點**（標題/列表/表格/程式碼塊/加粗，連續 8 行純散文即灰牆）、
**資訊形態決定 Markdown 元素**（並列→列表、對比→表格、步驟→有序列表、
命令→程式碼塊，選型表見 references）、**加粗是稀缺資源**（每段 ≤2 處）。
講解採**漸進式**：金字塔（結論先行）+ 先具象後抽象（例子先於定義）+
一屏新概念 ≤3 個。完整規範見 `references/markdown-typography.md`。

### 6. 第一屏一段話摘要（文件級 front-loading）

文件開頭（術語表之前）用 3-4 句話回答：這是什麼、跟讀者有什麼關係、
讀者需要做什麼、什麼時候生效。掃讀者只讀第一屏——第一屏沒有答案的文件，
對 80% 的讀者等於沒寫。checklist 裡的 front-loading 是段落級的，這條是
文件級的，兩者都要。

### 7. 正式語氣

去口語（「大家」「卡半天」「人肉」「別人才發現」）、去營銷腔（「核心 / 閉環 / 兜底 / 單一來源 / 守護」）、去口號收束（「靠人換靠機制」）、去對講者旁白（「宣講時不要求記憶命令」）。稱呼統一「成員 / 同學」。

### 8. 數字用模糊量

易漂移的小數字（測試數、行數、cell 數）用「200+」「manifest 矩陣」這種模糊量，不寫死「274」「28-cell」——過兩天加一個就過時。穩定的數字（40 個 skills、8 個 submodule）可精確。

## 可讀性 9 維度自檢

寫完後逐項打分（1-5），目標 ≥ 34/45：

1. **句子長度**：單句 ≤ 25 字，長句拆 2-3 句。
2. **段落密度**：段落 ≤ 5 行，一段一要點。
3. **資訊層級**：標題反映邏輯結構，泛標題（「新增能力」「使用方式」）加資訊量。
4. **掃讀性**：5 秒能抓要點，關鍵資訊前置（front-loading），善用加粗。
5. **列表與表格**：並列資訊結構化，不擠在散文裡。
6. **術語密度與解釋**：首現術語有解釋，單位文字術語量不超載。
7. **過渡與連貫**：章節順序符合「背景→問題→方案→操作」，段間有過渡。
8. **冗餘與重複**：同一資訊不重複，廢話湊數刪掉。
9. **版式與視覺錨點**：無灰牆屏（每屏有錨點）、元素選型對路、標題節奏
   均勻、TL;DR 在第一屏。

詳見 `references/readability-checklist.md`。

## 工作流

1. **摸清要講什麼**——讀 diff / 問使用者，確認核心內容。
2. **列大綱 + 屏分佈規劃**——按骨架模板列節（每節一句話要點），同時預估
   每節幾屏、每屏的視覺錨點是什麼（`references/markdown-typography.md` §3）。
3. **寫初稿**——按核心原則寫，術語表前置。
4. **自檢去重**——同一斷言只在最合適處詳述，其他交叉引用。
5. **可讀性 9 維度自檢**——逐項打分，<4 的維度改。注意：**自評分數普遍
   虛高一檔**（作者 agent 給自己打 4-5 是常態），自檢只用來抓明顯問題，
   分數以第 7 步獨立複評為準。
6. **零上下文 agent 評審**——起 2-3 個不同背景的 agent（產品/QA/新人）只讀
   文件報告看不懂的地方，據此改。**對外發布、跨職能宣講、或 800 字以上的
   文件必跑**；只有短小的內部速記可跳過。視角與 prompt 模板見
   `references/zero-context-review-prompts.md`（佔位符換成目標工程的實際
   子系統名）。
7. **可讀性 agent 複評**——獨立 agent 按 9 維度打分做迴歸驗證。
8. **收斂即停**（迭代有預算）：複評 ≥ 34/45 且三方評審無「完全看不懂」級
   問題 → 定稿；最多兩輪修訂，仍有殘留的小問題列成文末「已知取捨」而不是
   無限打磨——文件的邊際收益在兩輪後急劇下降。

## 關鍵避免

- **不要假設讀者有上下文**——任何前提先鋪墊
- **不要重複同一斷言**——去重，交叉引用
- **不要堆術語**——首現解釋，術語表集中
- **不要長段落**——拆分、結構化
- **不要口語 / 營銷腔**——正式語氣
- **不要寫死易漂移數字**——用 200+
- **不要對講者旁白**——文件是給讀者看的，不是描述宣講會議
- **不要灰牆**——連續 8 行以上純散文必須重排；每屏要有視覺錨點

## 骨架模板（列大綱時的預設起點）

```markdown
# <標題：讀者視角的一句話，不是工程視角>

<第一屏摘要：這是什麼 / 跟你有什麼關係 / 你要做什麼 / 什麼時候生效>

## 術語表          ← 貫穿全文的術語，一表定義
## 背景            ← 讀者需要的最小前情，不是專案史
## 問題            ← 讀者能感知的痛，不是內部技術債清單
## 方案            ← 做了什麼改變（對照表：此前/現）
## 你要做什麼      ← 行動指令前置到具體：命令/步驟/截止時間
## 效果與驗證      ← 怎麼確認自己弄對了
## FAQ / 已知取捨
```

按需增刪小節，但「第一屏摘要」與「你要做什麼」不可省——前者服務掃讀者，
後者服務成功動作（原則 0）。範例文件留在各自的源工程裡；帶走本 skill 時
不依賴任何外部路徑。
