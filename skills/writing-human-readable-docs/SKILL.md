# Writing Human-Readable Docs（写人类可读的技术文档）

## 目标

写出「零上下文听众也能看懂」的技术文档——宣讲稿、升级说明、公告、上手指南。核心判据：一个刚加入的新人或跨职能同事（产品/QA），不读仓内任何其他文件，只凭这一份文档，能不能看懂。

## 何时触发

- 用户说「写个宣讲文档 / 升级说明 / briefing / 公告」
- 用户反馈「文档像天书 / 看不懂 / 语气怪 / 太复杂」
- 用户要 review 一份已有文档的可读性
- 写给跨职能听众（不只工程师）的任何技术说明

## 核心原则（按重要性排序）

### 1. 零上下文假设（最关键）

假设读者不知道任何背景：不认识仓里的子仓、不懂 harness / agent-kit 是什么、不知道团队在用 AI 客户端写代码。所有前提先铺垫，再用。

反面：直接抛「274 个测试针对 harness 本身」——读者不知道 harness 是什么、274 是什么。
正面：先说「团队用 AI 客户端写代码，工作台是团队的工作区，harness 指工作台外层的脚手架脚本」，再说 274。

### 2. 术语表前置

贯穿全文的术语，在文档开头集中定义一次（一个表格），后续直接用，不重复解释。

反面：submodule 在 3 处各解释一遍「git 的子仓库功能…」。
正面：开头术语表定义一次，正文只用「submodule」。

### 3. 去重——同一信息只详述一处

同一断言（如「测试边界」「签名两条路径」）只在最合适的一节详述，其他处用「见 §X」交叉引用。

反面：签名两条路径在 §2.4、§3、§5.6、§6.1、§10 FAQ 五处各写一遍。
正面：§6.1（使用处）详述，其他处「见 §6.1」。

### 4. 结构化优于散文

并列信息用列表或表格；「此前 / 现」对比用加粗分点；案例用对称结构（场景/问题/正解）；变化总览用对照表。

反面：一段 150 字散文塞「场景 + 旧做法 + 问题 + 新方案 + 原因」。
正面：拆成「- 场景 / - 问题 / - 正解」三行。

### 5. 正式语气

去口语（「大家」「卡半天」「人肉」「别人才发现」）、去营销腔（「核心 / 闭环 / 兜底 / 单一来源 / 守护」）、去口号收束（「靠人换靠机制」）、去对讲者旁白（「宣讲时不要求记忆命令」）。称呼统一「成员 / 同学」。

### 6. 数字用模糊量

易漂移的小数字（测试数、行数、cell 数）用「200+」「manifest 矩阵」这种模糊量，不写死「274」「28-cell」——过两天加一个就过时。稳定的数字（40 个 skills、8 个 submodule）可精确。

## 可读性 8 维度自检

写完后逐项打分（1-5），目标 ≥ 30/40：

1. **句子长度**：单句 ≤ 25 字，长句拆 2-3 句。
2. **段落密度**：段落 ≤ 5 行，一段一要点。
3. **信息层级**：标题反映逻辑结构，泛标题（「新增能力」「使用方式」）加信息量。
4. **扫读性**：5 秒能抓要点，关键信息前置（front-loading），善用加粗。
5. **列表与表格**：并列信息结构化，不挤在散文里。
6. **术语密度与解释**：首现术语有解释，单位文字术语量不超载。
7. **过渡与连贯**：章节顺序符合「背景→问题→方案→操作」，段间有过渡。
8. **冗余与重复**：同一信息不重复，废话凑数删掉。

详见 `references/readability-checklist.md`。

## 工作流

1. **摸清要讲什么**——读 diff / 问用户，确认核心内容。
2. **列大纲**——背景→问题→方案→使用→效果→行动，每节一句话要点。
3. **写初稿**——按核心原则写，术语表前置。
4. **自检去重**——同一断言只在最合适处详述，其他交叉引用。
5. **可读性 8 维度自检**——逐项打分，<4 的维度改。
6. **零上下文 agent 评审**（可选但推荐）——起 2-3 个不同背景的 agent（产品/QA/新人）只读文档报告看不懂的地方，据此改。
7. **可读性 agent 复评**——回归验证分数提升。
8. **定稿**。

## 关键避免

- **不要假设读者有上下文**——任何前提先铺垫
- **不要重复同一断言**——去重，交叉引用
- **不要堆术语**——首现解释，术语表集中
- **不要长段落**——拆分、结构化
- **不要口语 / 营销腔**——正式语气
- **不要写死易漂移数字**——用 200+
- **不要对讲者旁白**——文档是给读者看的，不是描述宣讲会议
