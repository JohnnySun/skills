# 三檔模型共用一份 system prompt 的問題與解法

## 先說結論

不要再用「一份 prompt 打三個模型」。根因是：**同一段指令對不同能力層級的模型負載不同**——便宜模型需要「更多腳手架」（顯式格式規範、範例、強制檢查點）才能穩定輸出；旗艦模型天生指令遵循力強，同一段為了防便宜模型跑飛而堆的約束和範例，對旗艦模型反而變成「你話很多，我就順著多想多寫」的誘因，導致囉嗦。你不能用同一份文字同時滿足這兩種相反的需求，所以要拆。

## 具體怎麼拆：共用核心 + 分層差異化，不是三份完全獨立的 prompt

維護三份互相獨立的 prompt 是災難（业务规则改一次要改三處，三處還會慢慢漂移）。正確結構是：

```
[SHARED CORE]           ← 業務邏輯、摘要範圍定義、禁止事項，三檔共用，改一次全生效
  +
[TIER OVERLAY]          ← 只放「為了讓這個能力層級的模型穩定輸出」需要的東西
  = 最終送給對應模型的 prompt
```

實作上就是一個函式：`buildSummaryPrompt(tier)`，SHARED_CORE 是一個常數字串，三個 tier overlay 各自是一小段，拼接起來。改業務邏輯只改 SHARED_CORE，改某一檔的輸出穩定性只改對應 overlay。

### SHARED_CORE（三檔都要有，這段解決「說什麼」）

```
你是摘要助手。任務：把使用者提供的內容濃縮成一段摘要。

摘要必須：
- 保留原文的關鍵事實、數字、結論
- 不添加原文沒有的資訊或個人推測
- 使用與原文相同的語言

摘要禁止：
- 加入「以下是摘要：」「總結來說」這類引導語
- 復述你收到的指令內容
```

這段解決「摘要該長什麼樣」，跟模型能力無關，三檔都一樣。

### 便宜開源模型 overlay（解決「跑飛」——問題是指令遵循力弱，要用結構替代理解）

便宜模型跑飛通常不是不懂任務，是**在長 prompt 裡抓不住哪條約束優先、格式要求會被目標內容擠掉**。對策是把「自然語言描述格式」換成「結構化範例 + 顯式邊界」：

```
【輸出格式，嚴格遵守】
只輸出以下 JSON，不要有任何其他文字、不要用 ```json 包裹：
{"summary": "<這裡放摘要文字，一個字串，不換行>"}

【範例】
輸入：今天會議決定third-party API改用方案B，預計下週三上線，成本比方案A低20%。
輸出：{"summary": "會議決定改用方案B，下週三上線，成本比方案A低20%"}

【自我檢查，輸出前確認】
1. 我的輸出是不是只有一個 JSON object？
2. JSON 能不能被標準解析器解析（沒有多餘逗號、沒有未轉義的引號）？
3. summary 欄位是不是純文字，沒有 markdown？
```

三個關鍵手段，缺一不可：
1. **給範例，不要只給格式描述**——便宜模型對「照著這個例子做」的服從度遠高於對「請輸出JSON格式」這種抽象指令的服從度。
2. **用 JSON schema/結構化輸出**而不是自然語言段落——如果你的推理服務支援 `response_format: json_schema` 或等效的 constrained decoding，直接開，比純靠 prompt 約束可靠一個數量級，這是本質解法，不只是「prompt 寫更細」。
3. **加一段「自我檢查」清單**——便宜模型在生成末尾重新核對格式，實測能顯著降低格式錯誤率（相當於免費的一次自我修正）。

如果格式要求復雜（多欄位、巢狀），便宜模型光靠 prompt 仍可能不穩，這時候在應用層加**輸出校驗 + 失敗重試**：校驗失敗就把「錯誤在哪裡」拼回 prompt 重問一次，而不是指望一次 prompt 就打滿分。這是給便宜模型的安全網，旗艦模型基本用不到。

### 旗艦模型 overlay（解決「囉嗦」——問題是約束太多、旗艦會過度回應）

旗艦模型的問題往往不是不會簡潔，是 prompt 裡那些為了防便宜模型跑飛而寫的範例、格式規範、自我檢查清單，被旗艦模型當成「這是一個需要謹慎、詳盡處理的任務」的訊號，於是它認真地把每一條都展開解釋一遍。解法是**拿掉多餘腳手架、加明確的長度/風格硬約束**：

```
【長度與風格，硬性限制】
- 摘要最多 2 句話，不超過 60 個字
- 不要解釋你的摘要邏輯、不要列點、不要加任何前言或結語
- 只回傳摘要本文，不要有其他文字
```

不要指望「請簡潔」這種軟性詞——旗艦模型對「簡潔」的理解彈性很大，等於沒有約束。要給**可驗證的硬指標**（字數/句數上限），這是唯一對旗艦模型可靠的煞車。同時把便宜模型 overlay 裡的 JSON 範例、自我檢查清單整段拿掉——旗艦模型不需要靠範例學格式，你的格式指令一句話說清楚就夠，多餘的範例只會被當成「這任務比想像中複雜」的暗示。

### 主力模型 overlay（中間檔，通常最簡單）

主力模型能力介於兩者之間，overlay 通常只需要一句話級別的格式提醒 + 適度的長度建議，不需要 JSON 範例那麼重的腳手架，也不需要旗艦那麼硬的字數上限（可以用「1-3 句」這種較鬆的建議）。實務上很多團隊發現主力檔的 overlay 幾乎是空的——SHARED_CORE 對它已經足夠，這也是一個訊號：如果主力檔穩定，說明問題確實出在兩端各自的極端需求上，而不是核心描述本身有問題。

## 落地順序（TDD 角度，別一次全改）

1. 先找出便宜模型當前實際跑飛的樣本（收集 10-20 個真實跑飛輸出），確認失敗模式是「格式錯」還是「內容也跑飛」——這決定你要不要上 JSON schema constrained decoding。
2. 針對便宜模型先寫失敗案例對應的校驗測試（Red：目前輸出不符 schema），再套上面的 overlay + 開 structured output，跑到 Green。
3. 旗艦模型同理，先量化「囉嗦」（比如摘要平均字數 vs 目標字數），加硬約束後回歸驗證字數是否收斂。
4. 三檔都跑一輪你現有的摘要品質評測集，確認 SHARED_CORE 抽出來以後三檔的摘要語義沒有跑掉（避免拆分時不小心把某條業務規則漏到某個 overlay 裡）。

## 一個常見誤區要避開

不要用「模型能力強就加 prompt，能力弱就砍 prompt」這種反直覺但很多人會踩的方向——應該反過來：**能力弱的模型需要更多外部結構（範例、schema、校驗），能力強的模型需要更少指令噪音、更多硬約束**。腳手架的方向對便宜模型是「加」，對旗艦模型是「拆」，這是本回答的核心分歧點，兩者調換過來就是你現在遇到的問題本身。
