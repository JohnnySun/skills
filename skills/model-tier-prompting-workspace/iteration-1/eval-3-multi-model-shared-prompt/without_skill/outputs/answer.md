完整回答如下。

## 問題根因

一段 prompt 要同時伺候三個能力差很多的模型，必然兩頭不討好：為了壓住便宜模型的格式問題而加的細節規範，旗艦模型會過度遵循、輸出變囉嗦；為了讓旗艦自然發揮而放寬的部分，便宜模型就跑飛。**建議停止共用單一 prompt，改成「共用核心 + 按模型分層的附加段」**。

## 具體做法

把 system prompt 拆成兩部分組裝：

**共用核心（所有模型相同）**：任務定義、摘要的內容要求（保留關鍵事實、人名、數字、結論）、語言與立場要求。這部分是產品邏輯，跟模型無關。

**模型附加段（按檔位選一段拼上）**：

給開源便宜模型的附加段——重點是把格式釘死：
```
輸出格式（嚴格遵守）：
- 只輸出摘要本身，不要開場白、不要結尾評論、不要解釋你做了什麼
- 摘要為 2-4 句、單一段落、不用列點
範例：
輸入：<一段 300 字的會議記錄>
輸出：團隊確認 Q3 上線日期為 9/15，主要風險是支付接口聯調延期；由 Alice 負責在下週前給出備援方案。
```
如果你們的 serving 框架支援（vLLM/TGI 都支援），直接上 guided decoding / JSON schema 約束輸出，比任何 prompt 措辭都可靠。

給旗艦的附加段——重點是砍長度、給正面樣本：
```
輸出 2-4 句的摘要。直接給結論性內容，省略過程性描述與鋪墊。
好的範例：<一句示範>
```
旗艦不需要格式範例堆疊，給它一個「好的樣子」+ 明確長度上限就夠；也可以看 API 是否有輸出長度/詳盡度參數可以調，比在 prompt 裡反覆強調更穩定。

主力檔通常共用核心就夠，先不加附加段，觀察一段時間再說。

## 工程實作

- 附加段跟模型檔位綁定存配置（model_tier → prompt_suffix 的映射表），不要寫死在代碼裡，之後換模型只動配置。
- 每個檔位準備 5-10 個固定測試輸入，改 prompt 前後各跑一遍對比（格式合規率、平均長度），不要憑感覺上線。
- 監控兩個簡單指標：便宜模型的「格式不合規率」（好偵測：開頭是不是廢話、是否超長）、旗艦的平均輸出長度。有數字才知道改動有沒有效。
