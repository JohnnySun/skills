# 診斷：這不是「哪個模型比較笨」的問題，是「一份提示服務三層模型」的架構問題

你描述的兩個症狀（便宜模型格式跑飛、旗艦模型囉嗦）表面看是兩個獨立 bug，
其實是同一個根因：**三檔模型缺的東西不一樣，你只給了一份文本**。

- 便宜模型缺的是「過程與格式」——它需要被明確告知每一步怎麼做、輸出長
  什麼樣子。共用 prompt 為了不冒犯旗艦模型，通常寫得比較「意圖式」（一句
  話描述要做什麼），這對旗艦模型剛好，但便宜模型拿到這種模糊指令時，
  格式細節全靠自己猜——猜錯就是你看到的「跑飛」。
- 旗艦模型缺的是「意圖、邊界、停止條件」，不缺過程指導。共用 prompt 如果
  為了讓便宜模型不跑飛而塞了步驟清單、格式範例、"仔細思考每一步" 這類
  過程腳手架，旗艦模型會老實地把這些步驟都做一遍、每步都寫一段——這就是
  你看到的「囉嗦」。腳手架對強模型是負優化，不是保險。

換句話說：讓便宜模型變準的手段，正是讓旗艦模型變囉嗦的手段。共用同一份
文本，兩邊不可能同時滿意，這不是靠微調用詞能解決的，要換架構。

## 解法：骨架共用、厚度分層

不要維護三份互相獨立的 prompt（版本漂移、改一處要記得改三處），也不要
硬撐一份萬用文本。做法是拆成兩層：

1. **共用骨架**（所有檔位都吃這段，只維護一份）：任務定義（輸入是什麼、
   輸出是摘要）、輸出契約（用戶會看到的欄位/格式規格）、邊界（不要新增
   原文沒有的資訊、不要加入評論性判斷）、驗收標準（涵蓋所有關鍵資訊、
   符合格式、符合長度)。
2. **分層厚度**（依模型檔位開關，各自獨立維護，改一層不影響另外兩層）：
   - 頂級旗艦：補「為什麼」與「留白權」，砍掉任何過程指導。
   - 主力檔：補「明確範圍」與「正面範例」校準風格，不加步驟清單。
   - 經濟檔：補步驟清單、逐欄位輸出契約、few-shot 範例、明確停止條件，
     並且盡量把「格式」這件事從提示文字裡拿出來，改用 structured output
     / JSON schema 強制（如果供應商支援），把驗證放到程式碼外部而不是
     信任模型自我檢查。

實作上，工程上建議做成「模板 + 變數」：一個共用 skeleton 模板 + 一個依
`model_tier` 參數選擇的 addendum 區塊，兩者在程式碼裡拼接，而不是三份
完全獨立的字串常數——這樣共用骨架改一次，三檔都同步更新，只有分層差異
需要各自維護。

## 具體 prompt 產出

### 共用骨架（三檔都吃這段，固定不變）

```
<task>
你的任務是把使用者提供的內容（對話記錄 / 文章 / 會議紀錄）整理成摘要。
摘要的用途是讓讀者在不看原文的情況下，快速掌握關鍵資訊與結論。
</task>

<output_contract>
輸出格式：
{
  "title": "一句話標題，概括本次內容主旨",
  "key_points": ["條列重點，每條一句話，3-6 條"],
  "summary": "2-4 句話的完整摘要段落"
}
只輸出這個 JSON 物件，不要輸出其他文字。
</output_contract>

<boundaries>
只整理原文出現過的資訊，不要補充原文沒有的推論、評價或建議。
如果原文本身內容不足以支撐 3 條重點，key_points 可以少於 3 條，不要湊數。
</boundaries>
```

### 頂級旗艦檔位追加（例：Claude Fable/Mythos 級、GPT 系下一代旗艦）

```
<addendum tier="frontier">
這份摘要會被用戶直接看到，用來決定要不要回頭看完整內容——所以精準比
完整更重要，抓不準重點比漏掉細節更傷。

長度與詳略程度由你根據原文的資訊密度判斷；不需要固定湊到某個字數或
條數。以下是一個「長度剛好」的範例，供你校準風格基準（不是要你套用
這個字數，是要你感受這種精煉程度）：

[範例輸出，2-4 句 summary、3-4 條 key_points，每條一句話不展開]

不需要展示你的整理過程或逐步推理，直接輸出最終結果。
</addendum>
```

要點：這裡刻意不寫步驟清單、不寫「先讀一遍再整理」這類過程指導，也不
用「不要囉嗦」這種負面指令（definitionally 不可操作），改用一個正面
範例讓模型自己校準「多長算剛好」。這是回應 anti-pattern「只寫負面指令」
與「CoT 腳手架塞給強模型」——旗艦囉嗦的常見成因往往就是這兩條疊加。

### 主力檔位追加（例：Claude Opus/Sonnet 級、GPT-5.x 主力）

```
<addendum tier="workhorse">
適用範圍：以下規則套用到輸入內容的**每一段**，不只是開頭或看起來最重要
的部分——如果原文有多個子主題，key_points 要覆蓋到每個子主題，不要只
挑你覺得最突出的一個。

摘要語氣：中性陳述，不加情緒詞、不加評價形容詞（例如不要用「令人驚訝
的是」「值得注意的是」這類引導性措辭）。

範例（風格校準用）：
[一個具體的輸入→輸出範例，長度中等偏精煉]
</addendum>
```

要點：主力檔位的特徵是「字面遵循但不自動推廣」，所以「涵蓋每個子主題」
這種範圍要求必須寫明,不能假設它會自己推廣。這層不需要步驟清單，一個
正面範例加明確範圍就夠。

### 經濟檔位追加（例：Haiku 級、GLM 等開源主流、mini/flash 檔）

```
<addendum tier="economy">
請嚴格按以下步驟操作：
1. 通讀輸入內容一次。
2. 用一句話寫出 title，概括整體主旨（不要用問句，不要用省略號結尾）。
3. 列出 3-6 條 key_points，每條必須是完整的一句話陳述,不要用「等等」
   「以及其他」這類省略詞收尾。
4. 寫一段 2-4 句的 summary，只能使用原文出現過的資訊。
5. 檢查輸出是否為合法 JSON，欄位順序為 title → key_points → summary，
   不要有多餘欄位，不要在 JSON 前後加任何說明文字。

以下是兩個完整範例，包含一個「輸入內容較短、key_points 少於 3 條」的
邊界情況：

範例 1（一般情況）：
輸入：[...]
輸出：{"title": "...", "key_points": [...], "summary": "..."}

範例 2（邊界情況：輸入內容不足以撐出 3 條重點）：
輸入：[...]
輸出：{"title": "...", "key_points": ["...", "..."], "summary": "..."}

完成以上 JSON 輸出後立即停止，不要附加任何解釋、免責聲明或後續問句。
</addendum>
```

要點：這一層是傳統提示工程（步驟清單、格式契約、few-shot、明確停止
條件）真正生效的地方,不要省。同時工程上建議兩個配套動作，這兩個比
提示文字本身更能治本：

1. **能力允許的話,格式驗證交給結構化輸出而不是提示文字**：如果這個
   開源模型的部署方式支援 function calling / JSON schema 約束輸出
   （例如 vLLM 的 guided decoding、或模型本身支援 tool use 格式），
   優先用這個機制鎖死欄位結構,不要只靠提示文字描述格式——提示文字
   對這層模型是「建議」,結構化輸出約束才是「強制」。
2. **驗證外置,不要信任這層模型自我檢查**：拿到輸出後,程式碼層面做
   JSON schema 驗證;驗證失敗時,重新請求一次並把錯誤訊息（例如「上次
   輸出缺少 key_points 欄位」）附進下一輪提示,而不是靜默丟給用戶一個
   壞掉的摘要,也不要指望在提示裡加一句「請確保輸出正確格式」就能解決
   ——這層模型的自我驗證不可靠。

## 驗證方式（別跳過這步）

改完不能只憑感覺說「應該有解決」。三檔各自要有一組真實案例（至少
3-5 組，涵蓋長文/短文/多子主題/單一主題這幾種輸入形態）：

- 便宜模型：改動前後各跑一輪，檢查輸出 JSON 是否能被程式解析（parse
  成功率）、欄位是否齊全,這是量化的、有 pass/fail 的驗收,不是「感覺
  有沒有好一點」。
- 旗艦模型：改動前後對比輸出長度與資訊密度,盲評兩個版本哪個更貼近
  「摘要」而不是「複述」。
- 若這個摘要功能本身有線上留存的 eval 集或人工評分流程,拿這三版分層
  提示各跑一次那個既有 eval,不要只憑這次的臨時案例下結論。

## 一句話結論

不是「這句話要換個說法」的問題,是「三檔模型需要三份不同厚度的
addendum,疊在同一份共用骨架上」——共用骨架維護任務定義與輸出契約,
分層 addendum 各自維護該檔位真正缺的東西（旗艦缺邊界與留白權、主力缺
明確範圍、經濟檔缺步驟與格式強制),而不是三檔共用同一份「怕得罪誰就
各退一步」的折衷文本。
