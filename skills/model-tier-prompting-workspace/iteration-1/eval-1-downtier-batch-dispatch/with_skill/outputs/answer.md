# 回答

## 先歸層（動筆前兩問）

- **誰執行**：你說「最便宜的模型」= **快速經濟層**（Haiku 級 / GLM 等開源主流 / 各家 mini-flash）。這層缺的是「過程與格式」，不是「意圖」——所以派工 prompt 要反過來加厚，不能用你我平常互相溝通的三句話意圖式寫法。
- **在哪執行**：這是**一次性 headless 子代理**（派出去、跑完、拿回最終訊息，沒有追問機會）。最終訊息就是你能拿到的全部，沒寫輸出契約的話大概率只會拿回「我完成了」。

因此下面這份 prompt 按「經濟層 + headless 一次性派工」的骨架寫：步驟清單、輸出格式範例、分支判斷全部前置、明確停止條件、驗證外置（不依賴子代理自查真假）。

## 我替你做了 3 個假設（若不符請先改 prompt 裡對應段落再派工）

1. **目標 schema 長相**：假設現有 frontmatter 是
   ```yaml
   ---
   title: 範例標題
   tags: 前端, 效能, uni-app
   ---
   ```
   要變成
   ```yaml
   ---
   title: 範例標題
   tags: [前端, 效能, uni-app]
   updated: 2026-07-08
   ---
   ```
   即 `tags` 改成 YAML flow sequence（`[a, b, c]`），`updated` 用 `YYYY-MM-DD`。如果你們的目標格式是區塊列表（`tags:\n  - a\n  - b`）或 `updated` 要用 ISO 時間戳，先改 prompt 裡的「範例」段落——經濟層模型完全依賴這個範例，範例錯了 40 個檔案會整批錯。
2. **`updated` 欄位的值從哪來**：沒有讓子代理「自己填今天日期」或「自己判斷」——這種開放式判斷丟給經濟層模型只會拿到牠的第一直覺（通常是執行當下的系統日期，跟檔案實際最後修改時間無關，且不可重現/不可驗證）。改成用 `git log -1 --format=%ad --date=short -- <file>` 取該檔案的最後一次 commit 日期，確定性、可驗證、任何人重跑都拿到同一個值。
3. **判斷用寫轉換腳本，不用逐檔手改**：40 個檔案讓經濟層模型一個一個用 Edit 工具改，格式漂移機率隨檔案數線性上升（這層的格式漂移是已知弱點，見 anti-patterns）。改成「寫一支小腳本跑一次」，把「40 次獨立判斷」壓成「一次腳本邏輯 + 一次驗證」，大幅降低出錯面。

如果這 3 條裡有跟你們實際情況不符的，先改下面 prompt 對應段落再派工——不要讓子代理現場猜。

## 派工 Prompt（可直接複製派給便宜模型，例如 Claude Haiku 4.5）

```
任務：把 docs/ 目錄下所有 markdown 檔案的 YAML frontmatter 統一成新 schema——
新增 updated 欄位、把 tags 從逗號分隔字串改成陣列。只改 frontmatter 區塊
（開頭 --- 到下一個 --- 之間），不改任何 frontmatter 以外的正文內容。

上下文：
- 目標目錄：docs/（相對於 repo 根目錄）
- 現有 frontmatter 範例：
  ---
  title: 範例標題
  tags: 前端, 效能, uni-app
  ---
- 目標 frontmatter 範例（這是唯一正確的輸出格式，逐字對照，不要自創格式）：
  ---
  title: 範例標題
  tags: [前端, 效能, uni-app]
  updated: 2026-07-08
  ---
- updated 欄位的值 = 該檔案最後一次 git commit 的日期，用這個命令取得：
  git log -1 --format=%ad --date=short -- <檔案路徑>
  不要用系統當前日期，不要自己猜測日期。若某檔案完全沒有 git 歷史（新檔案，
  上面命令輸出空字串），改用今天的日期 2026-07-08，並在報告中列出這些檔案。

步驟：
1. 先寫一支轉換腳本（Python 或 Node 皆可，你自行選擇），不要逐檔手動編輯。
   腳本邏輯：
   a. 列出 docs/ 目錄下所有 .md 檔案（含子目錄，遞迴）。
   b. 對每個檔案，讀取內容，判斷開頭是否為 frontmatter（第一行是 ---，
      且能找到第二個 --- 作為結尾）。
   c. 若沒有 frontmatter（開頭不是 ---，或找不到結尾 ---）：跳過該檔案，
      不修改，記錄到「跳過清單」。
   d. 若有 frontmatter，解析出其中的 tags 欄位：
      - 若 tags 已經是陣列格式（例如 tags: [a, b] 或區塊列表 tags:\n  - a）：
        保留原樣不動，但仍需檢查/新增 updated 欄位（見下）。記錄到
        「tags 已是陣列，僅補 updated」清單。
      - 若 tags 是逗號分隔字串（例如 tags: a, b, c）：拆分每個逗號分隔項，
        去除前後空白，轉成 tags: [a, b, c] 這種 flow sequence 格式。
      - 若沒有 tags 欄位：不新增 tags 欄位，只處理 updated；記錄到
        「無 tags 欄位」清單。
   e. 檢查是否已有 updated 欄位：
      - 若已存在：不覆蓋，保留原值，記錄到「已有 updated，未覆蓋」清單。
      - 若不存在：執行 git log -1 --format=%ad --date=short -- <檔案路徑>
        取得日期並新增 updated 欄位（加在 frontmatter 最後一行，緊接
        tags 之後）。若命令輸出為空，用 2026-07-08 代替，並記錄到
        「無 git 歷史，用預設日期」清單。
   f. 其餘所有 frontmatter 欄位（title 及其他既有欄位）原封不動，欄位
      原本的順序不變，只在 tags 之後插入/保留 updated。
   g. frontmatter 以外的正文內容一個字元都不要動。
2. 跑這支腳本，實際套用到 docs/ 目錄下所有檔案。
3. 跑完後，用下面這個驗證命令確認結果（貼出實際輸出）：
   grep -L "^updated:" docs/**/*.md 2>/dev/null || echo "全部檔案皆有 updated 欄位"
   以及至少挑 3 個被改過的檔案，貼出它們的 frontmatter 區塊原文，證明
   格式符合範例。
4. 不要修改 docs/ 目錄以外的任何檔案；不要修改 frontmatter 以外的正文；
   不要順手修正你發現的其他問題（錯字、格式、內容）——只做 frontmatter
   schema 統一這一件事。

停止條件：docs/ 目錄下所有 .md 檔案都檢查過一輪（無論是否需要修改）即
視為完成，不要重複跑、不要擴大到 docs/ 以外目錄。

你的最終訊息是本次任務唯一的交付物，之後沒有追問機會。它必須包含：
- 改動檔案清單（檔案路徑級，逐一列出被修改的檔案）
- 跳過清單（無 frontmatter 的檔案，逐一列出）
- 「tags 已是陣列，僅補 updated」清單
- 「無 tags 欄位」清單
- 「已有 updated，未覆蓋」清單
- 「無 git 歷史，用預設日期」清單
- 驗證結果：第 3 步驗證命令的實際輸出原文 + 3 個檔案的 frontmatter 原文
- 驗收標準逐條自查：
  1. 是否所有原本有 frontmatter 的檔案都已新增 updated 欄位？過/不過
  2. 是否所有原本 tags 為逗號字串的檔案都已轉成陣列格式？過/不過
  3. 是否所有 frontmatter 以外內容維持不變（可用 git diff 只顯示
     frontmatter 區塊行來佐證）？過/不過
  4. 是否只改了 docs/ 目錄下的檔案？過/不過
- 未解決事項與你建議的下一步（沒有就寫「無」）
```

## 為什麼這樣寫（給你，不是給子代理的部分）

- **步驟清單 + 範例 + 分支前置**：經濟層模型的已知弱點是「現場判斷淺、格式會漂移、自我驗證不可信」。上面 prompt 把「tags 已是陣列怎麼辦」「沒有 tags 怎麼辦」「已有 updated 怎麼辦」「沒 git 歷史怎麼辦」這四個可預見分支全部在 prompt 裡給了處置方式，不留給牠現場判斷（對應 anti-patterns 的「給經濟層留判斷空間」條目）。
- **改成寫腳本而不是逐檔編輯**：40 個檔案人工判斷 40 次，出錯機率遠高於「寫一次邏輯、跑一次、驗證結果」。這也讓「驗證外置」這條原則落地——驗收不依賴子代理口頭自查，而是有一個 grep 命令可以獨立重跑確認。
- **updated 值來源選 git log 而不是「今天日期」**：避免經濟層模型把「現在幾號」當成「這檔案何時更新」的替代品——兩者語義不同，且系統當前日期不可驗證、不可重現，git commit 日期可以。
- **輸出契約寫死清單結構**：headless 一次性派工沒有追問機會，這份契約保證你拿回來的不是「完成了」三個字，而是可以直接核對的清單，你不需要重新翻 40 個檔案去確認牠做對了沒。
- **沒有給「為什麼要做這件事」的大段背景**：經濟層上下文利用率低，多餘的背景敘事只會稀釋牠對步驟清單的注意力，所以我只留了任務直接需要的上下文（目錄、範例、命令）。
