# 派工 Prompt：docs/ frontmatter schema 批次遷移（40 檔）

## 先講判斷：這個任務適合下放到最便宜模型嗎？

適合，而且是下放到經濟層（Haiku 級）的典型正例：40 個檔案的改動是**同一條規則的機械重複**（加一個欄位、把一種字串格式轉成陣列），沒有跨檔案的語義判斷、沒有「這個檔案該不該改」的模糊地帶。這正是快速經濟層模型的強項——只要把「過程」講清楚，它不需要自己規劃路徑。

但正因為是經濟層執行，派工 prompt 不能用你我平常寫給主力/frontier 模型的「三句話意圖式」風格（那種風格對這層是資訊缺失，會導致漏改、格式改壞、或在遇到不規則輸入時瞎猜)。這層模型缺的是**過程與格式**，不是意圖——所以 prompt 要反過來加厚：顯式步驟、輸入輸出範例、每一種異常情況的處置方式都要在派工前想好、寫死，不能留給它「自行判斷」。

以下是可以直接派出去的完整 prompt。

---

## 派工 Prompt（可直接貼給子代理，執行模型指定為最便宜層級，如 Haiku）

```
任務：把 docs/ 目錄下所有 markdown 檔案的 YAML frontmatter 從舊 schema
遷移到新 schema。只改 frontmatter 區塊（檔案開頭 --- 到 --- 之間），不
改 frontmatter 以外的任何內容（正文、標題、程式碼區塊一律逐字保留）。

新 schema 的兩項變更：
1. 新增 `updated` 欄位（格式 YYYY-MM-DD）。
2. 把 `tags` 從逗號分隔字串改成 YAML 陣列。

## 範圍
- 只處理 docs/ 目錄下（含子目錄）副檔名為 .md 的檔案。
- 目標檔案清單：先執行 `find docs -name "*.md" -type f` 取得完整列表，
  逐一處理，不要憑記憶或猜測列表。
- 不創建新檔案，不刪除檔案，只做原地編輯。

## 步驟（每個檔案重複這個流程）
1. 讀取檔案，確認開頭是 `---`（frontmatter 起始標記）。
   - 如果檔案沒有 frontmatter（不是以 `---` 開頭）：**跳過此檔案**，
     記入報告的「跳過清單」，不要幫它新增 frontmatter。
2. 找到 frontmatter 的結束 `---`，只在這個區塊內操作。
3. 檢查是否已有 `updated` 欄位：
   - 已有 → 不動這個欄位（不要覆蓋既有值）。
   - 沒有 → 執行 `git log -1 --format=%cd --date=short -- <該檔案路徑>`
     取得這個檔案最後一次 commit 的日期，新增一行 `updated: <該日期>`。
   - 如果該指令沒有輸出（檔案還沒被 git 追蹤，例如新檔）：
     用今天的日期 `date +%F` 填入，並在報告中列出這種情況的檔案。
4. 檢查 `tags` 欄位：
   - 不存在 `tags` 欄位 → 不新增，跳過這一步，不要臆造 tags。
   - 已經是陣列格式（`tags: [a, b, c]` 或多行 `- ` 列表）→ 不動，視為
     已符合新 schema。
   - 是逗號分隔字串（例如 `tags: 設計, UI, 教學`）→ 轉成陣列格式
     `tags: [設計, UI, 教學]`：以逗號切分、每段前後空白去除，逐項放入
     陣列，不重新排序、不去重、不翻譯。
   - 格式無法判斷（例如看起來像陣列但語法錯誤、或值本身包含未跳脫的
     特殊字元）→ **不要自行修正**，跳過這個欄位的改動，在報告中列出
     原始內容供人工確認。
5. 只寫回 frontmatter 區塊有變動的檔案；沒有任何欄位需要變動的檔案，
   不要重寫它（避免產生無意義的 diff）。

## 格式範例

輸入（檔案原始 frontmatter）：
---
title: 角色卡設計指南
tags: 設計, UI, 教學
---

輸出（處理後）：
---
title: 角色卡設計指南
tags: [設計, UI, 教學]
updated: 2026-06-30
---

（2026-06-30 是範例：實際值用該檔案 git log 查出的日期，不要照抄這個
範例日期。）

## 停止條件
- 處理完 `find docs -name "*.md" -type f` 列出的全部檔案即算完成，不要
  主動擴大到 docs/ 以外的目錄。
- 遇到任何本 prompt 沒有覆蓋到的意外情況（例如檔案是二進位、frontmatter
  巢狀結構很複雜看不懂怎麼改），該檔案標記為「不確定，跳過」，不要憑
  自己的判斷硬改，寫進報告讓人工複查。

## 驗證（完成後必須執行，把原始輸出貼進報告）
1. `grep -L "^updated:" docs/**/*.md` 之類的檢查（或等價的逐檔 grep），
   確認所有「有 frontmatter 且被判定要加 updated」的檔案都確實加上了。
2. 對每一個你改過 tags 格式的檔案，跑一次 YAML 合法性檢查（例如
   `python3 -c "import yaml,sys; yaml.safe_load(open(sys.argv[1]).read().split('---')[1])" <file>`
   或倉庫裡既有的 frontmatter lint 工具，如果有的話優先用它）確認新
   frontmatter 是合法 YAML，不是被你手動拼字拼壞的。
3. 用 `git diff --stat` 附上一份改動檔案清單與行數統計。

## 最終訊息格式（你的最終訊息是本次任務唯一的交付物，之後沒有追問機會）
- 處理總數 / 實際改動數 / 跳過數（三個數字都要）
- 跳過清單：檔案路徑 + 跳過原因（逐條列，不要用「其他」含糊帶過）
- 「updated 欄位來源用了今天日期而非 git log」的檔案清單（如果有）
- 步驟 5 的驗證結果：三個驗證命令的原始輸出（不要只寫「驗證通過」）
- 未解決事項：有沒有需要人工複查的 tags 格式異常（沒有就寫「無」）
```

---

## 為什麼這樣寫（設計理由，供你檢查是否要調整）

- **步驟寫死、範例給了、每個分支的處置方式都前置**：這是經濟層 prompt 跟主力/frontier 層 prompt 最大的差異。像「tags 格式無法判斷」「檔案沒被 git 追蹤」這種岔路，如果不在派工前想好答案，經濟層模型現場會傾向編造一個看起來合理但不一定對的處理方式（例如亂猜日期、把奇怪的 tags 硬轉格式），事後很難追查是哪個檔案被錯誤處理。所以本 prompt 把「跳過」設成大多數異常情況的預設安全動作，而不是「自行判斷最佳方案」。
- **驗證放在外部、可執行、要求貼原始輸出**：沒有要求它「自己判斷做得好不好」，而是要求跑固定命令（grep / YAML 合法性檢查 / git diff --stat）並把命令輸出原文貼出來。這樣你收到報告後，不需要相信它的自我評價，看輸出就能判斷有沒有做對——這是把驗收這件事从「這層模型的自我判斷」移到「你能核對的證據」上。
- **只用「跳過 + 記錄」而不是「盡量修」處理不確定情況**：40 個檔案裡大概率會有 1-2 個不完全符合預期格式的 frontmatter（缺 tags、tags 已是陣列、根本沒 frontmatter）。經濟層模型硬要「自己想辦法」處理這些邊界情況風險最高、也最難事後稽核，所以策略是明確劃出「這些情況一律跳過並回報」，把最終判斷留給你或後續一個窄任務子代理去單獨處理那幾個檔案。
- **輸出契約寫在最後、要求具體數字與清單**：一次性子代理（headless）沒有追問機會，沒寫清楚輸出契約，最常見的結果就是回一句「已完成全部檔案的更新」——你完全無法知道它有沒有漏改、有沒有改壞。本 prompt 要求三個數字（處理/改動/跳過）+ 具體清單 + 驗證命令原始輸出，任何一項缺失都代表這次派工沒收好。

## 建議的執行方式

Agent 呼叫時把 `model` 參數設成最便宜層級（例如 `haiku`），並在 prompt 前段附上這份完整文本；不需要多輪往返——如果跑完發現有幾個檔案被跳過待人工複查，直接針對那幾個檔案另開一個新的窄任務子代理處理，而不是喚醒同一個子代理做第二輪（每次喚醒都要重載上下文，成本比重新派一個小任務還高）。
