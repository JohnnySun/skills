# 提示反模式清單

每條格式：症狀 → 為什麼有害（含受害層級）→ 怎麼修。審計既有提示
（skill、system prompt、review prompt、派工模板）時逐條掃。

## 1. 要求復述／展示內部推理

**症狀**：「先解釋你的思考過程」「把推理寫在回答開頭」「show your
reasoning」。
**為什麼**：頂級模型（frontier 層）上可能觸發 reasoning-extraction 類
refusal——這是硬故障，直接拿不到輸出；其他層級上則是純 token 浪費。
**修**：需要推理可見性時，讀結構化 thinking 輸出；需要向用戶展示過程時，
用專用的 send-to-user／進度工具。審計舊 skill 時把「反思並展示思考」類
指令全數清除。

## 2. 「只報高嚴重性問題」（召回率殺手）

**症狀**：review prompt 裡有「be conservative」「只報重要問題」「不要
吹毛求疵」，換代後召回率暴跌。
**為什麼**：新一代模型會**忠實執行**這條過濾——調查深度沒變，但把大量
真發現判為「低於你說的標準」而不報。測到的是框架效應，不是能力退化。
**修**：發現階段要覆蓋率不要過濾——「報告所有發現，含不確定與低嚴重性
的，附信心與嚴重度，由下游過濾」。真要單步自過濾，就給操作性標準
（「報告任何可能導致錯誤行為/測試失敗的問題；只略過純風格偏好」），
不用「重要」這種定性詞。

## 3. 強制進度節拍

**症狀**：「每 3 次工具呼叫後總結一次進度」。
**為什麼**：主力層以上已自帶校準良好的進度更新，強制節拍產生噪音、
打斷工作流。這是為舊模型補丁的遺留。
**修**：先移除再觀察；更新的長度/內容不合需求時，描述期望樣貌並給範例，
而不是規定頻率。

## 4. MUST/NEVER 堆疊代替解釋

**症狀**：連續多條全大寫禁令，沒有一條說為什麼。
**為什麼**：主力層以上有良好的意圖理解，解釋 why 能讓它把規則推廣到
你沒列舉的情況；純禁令只覆蓋字面情況，還會擠占上下文。寫到第三個
NEVER 是黃旗。
**修**：每條規則配一句動機。經濟層例外——它確實需要硬規則，但硬規則
也應少而精。

## 5. CoT 腳手架塞給強模型

**症狀**：給 frontier/主力模型的 prompt 裡有「Step 1: 列出關鍵概念；
Step 2: 逐一比對；Step 3: 排序前三……」。
**為什麼**：這是經典提示改進器（為弱模型設計）的簽名式輸出。強模型
自己的分解通常更好，寫死步驟會鎖死它繞開障礙的路徑，且過度規範可測地
壓低輸出品質。
**修**：改成意圖 + 驗收；步驟清單只保留給經濟層執行者。

## 6. 用提示繞參數

**症狀**：「think harder」「think very deeply before answering」貼在
prompt 裡，效果時好時壞。
**為什麼**：思考深度的正規旋鈕是 effort / reasoning 檔位；提示補償只在
被迫鎖低檔（延遲敏感）時才是正當手段，且效果遠不如升檔穩定。
**修**：推理太淺 → 升檔位。鎖低檔的場景才加一句針對性的「此任務涉及
多步推理，回答前仔細思考」。

## 7. 依賴 temperature 求多樣性

**症狀**：設計/文案類流程靠調 temperature 產生方案多樣性。
**為什麼**：新一代主力模型多數不再接受非預設採樣參數（直接 400）；
即使接受，改參數產生的也是噪音多樣性，不是方向多樣性。
**修**：改成「建構前先提出 N 個明確不同的方向（各附一行理由），讓
用戶選定後只實作那個」。

## 8. 顯示上下文餘量倒數

**症狀**：框架把剩餘 token 數展示給模型，模型開始建議「開新 session」
「先摘要交接」或自行縮減工作。
**為什麼**：倒數會誘發上下文焦慮行為，尤其長 session 尾部。
**修**：能不顯示就不顯示；必須顯示時補一句「你有充足上下文，不要因
上下文限制停止、摘要或建議新 session」。

## 9. 只寫負面指令

**症狀**：「不要太長」「不要用術語」「別過度解釋」，但沒有一個正面
樣本。
**為什麼**：負面指令只告訴模型排除什麼，沒定義目標分佈；各層級上，
展示期望輸出的正面範例都比負面清單有效。
**修**：給一段「這樣寫就對了」的範例，負面清單只留最容易踩的一兩條。

## 10. 定性詞當標準

**症狀**：「保守一點」「重要的才報」「適當簡潔」「高品質」。
**為什麼**：字面化的模型會按**它的**定義執行你的定性詞，結果不可預測
且不可調試——你不知道它把閾值放在哪。
**修**：翻譯成操作性標準：可觀察的行為、可枚舉的類別、可驗證的條件。

## 11. 一份提示服務所有模型

**症狀**：同一段長 prompt 不加條件地餵給 frontier 主力經濟三層（常見
於「模型可配置」的產品和多模型派工框架）。
**為什麼**：三層缺的東西不同（見 SKILL.md），一份文本必然對某層過度
規範、對另一層資訊不足。
**修**：骨架共用（任務/邊界/驗收/輸出契約），厚度分層——按目標層級
增減步驟清單、範例與 why。維護成本可控的做法是分層模板 + 共用變數。

## 12. 給經濟層留判斷空間

**症狀**：派給經濟層模型的 prompt 寫「遇到問題自行判斷處理」「你決定
最佳方案」。
**為什麼**：這層的現場判斷是淺層第一直覺，且不會回來問你；模糊處會被
編造填平。
**修**：分支判斷前置——可預見的異常（格式不符、檔案缺失、衝突）逐一
給處置方式；不可預見的，指令是「停下並在報告中描述」，不是「自行處理」。

## 13. 舊模型的風格補丁原樣搬運

**症狀**：換代遷移時，把上一代調教出來的語氣提示、前端美學長文、
verbosity 控制原封不動帶過去。
**為什麼**：每一代的預設基準都在漂移（更直接、少肯定開場、更強的
內建設計直覺）；舊補丁疊在新基準上會過矯正，甚至互相打架。
**修**：遷移時風格類提示全部重新答辯：先裸跑看新基準，有差距才補，
按新基準的差距補。
