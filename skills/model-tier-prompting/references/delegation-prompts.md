# 派工 Prompt：給子代理與跨模型委派寫提示

派工是**跨層級提示**最高頻的場景：主 session 往往跑在高層級模型，而施工
子代理按成本紀律派給更低層級——也就是說，寫派工 prompt 的模型和執行它的
模型不是同一層。寫的那方會不自覺用「自己讀得懂」的風格寫（簡短、意圖式、
省略過程），這對低層執行者恰好是資訊缺失。**寫派工 prompt 時，按執行者
的層級寫，不按自己的習慣寫。**

## 通用骨架（六段，層級決定各段厚度）

```
1. 任務：一句話說清楚要做什麼（動詞開頭，範圍名詞明確）。
2. 上下文：執行者不知道但需要知道的事——相關檔案路徑、已確立的決定、
   為什麼做這件事（一句 why）。
3. 邊界：明確不做什麼。相鄰的誘惑（順手重構、擴 scope）點名禁止。
4. 驗收標準：怎樣算完成。可驗證的形式（測試通過、輸出含 X、檔案存在）。
5. 輸出契約：最終訊息要包含什麼。headless 派工必寫（見下）。
6. 完成前自查：要求對照驗收標準逐條自查後才收尾。
```

## 一次性派工契約（headless 子代理必加）

子代理的最終訊息就是你拿到的全部——它不是給人看的聊天，是返回值。
沒寫輸出契約的派工，拿回來的常是「我完成了」三個字。模板：

```
你的最終訊息是本次任務唯一的交付物，之後沒有追問機會。它必須包含：
- 改動/產出清單（檔案路徑級）
- 驗證結果（跑了什麼命令、輸出關鍵行原文）
- 驗收標準逐條自查（過/不過/不適用+原因）
- 未解決事項與你建議的下一步（沒有就寫「無」）
```

為什麼用一次性而不是多輪往返：每次喚醒子代理都要重載上下文，幾輪往返的
開銷超過子代理本身。需要補充資訊時，寧可派一個新的窄任務子代理。

## 按執行者層級調整

### 派給 frontier / 主力層（意圖式）

給 why + 邊界 + 驗收，**不給步驟清單**——這層自己規劃的路徑通常比你
預先寫死的好，步驟清單反而鎖死它繞開障礙的空間。

```
任務：修復 <模組> 的 <症狀>。
背景：<一句 why——這影響誰、為什麼現在修>。已確認 <已排除的假設>。
邊界：只改 <範圍>；不重構無關代碼；發現相鄰 bug 記錄不修。
驗收：<測試命令> 全綠；<症狀的可觀察消失方式>。
最終訊息 = 完整交付報告（改動清單/驗證輸出/自查/遺留）。
```

補一條證據錨定（這層的長任務值得）：「報告中的每條完成聲明須對應你
實際執行過的工具結果；未驗證的部分明說未驗證」。

### 派給經濟層（過程式）

上面骨架全保留，再加厚：步驟清單、輸出格式範例、明確停止條件。

```
任務：把 <輸入> 轉換成 <輸出格式>。
步驟：
1. 讀 <檔案>，提取 <欄位>。
2. 按下方範例格式逐條轉換。
3. 寫入 <路徑>；只寫這個檔，不改其他檔。
4. 用 <驗證命令> 檢查，貼輸出。
範例：
輸入：<真實樣本一條>
輸出：<期望結果一條>
停止條件：完成 <N> 條或處理完輸入即停；遇到格式不符的輸入，跳過並
在報告中列出，不要自行修正。
最終訊息格式：
- 處理數 / 跳過數
- 驗證命令輸出
- 跳過清單
```

關鍵差異：**判斷點前置**。任何需要判斷的分支（格式不符怎麼辦、找不到
檔案怎麼辦）在 prompt 裡預先給出處置方式，不留給執行者現場判斷。

### 跨供應商：派給 GPT 系（區塊契約式）

GPT 系（含 Codex 類 agent）對穩定的 XML 區塊結構反應最好。像操作員
而不是協作者那樣下指令：緊湊、區塊化、一次一個任務。

```xml
<task>具體任務 + 相關 repo/失敗上下文</task>
<output_contract>最終輸出的精確形狀、順序、簡潔度要求</output_contract>
<follow_through_policy>預設繼續做而不是回來問的事項清單</follow_through_policy>
<verification_loop>實作/除錯/風險修復類必加：要跑什麼驗證、貼什麼證據</verification_loop>
<grounding_rules>review/研究類必加：每條結論要錨定到可引用的證據，猜測要標明</grounding_rules>
```

原則：優先收緊 prompt 契約，而不是提高 reasoning 檔位或加長自然語言
解釋；同一 thread 的跟進只送增量指令，不重述全文。

## 派工 prompt 自查清單

發出前過一遍：

1. ☐ 指定了執行模型（成本紀律）且 prompt 厚度與該模型層級匹配？
2. ☐ 有輸出契約？（headless 必有）
3. ☐ 邊界寫了「不做什麼」？（防 scope 蔓延，所有層級都要）
4. ☐ 驗收標準可被執行者自己驗證？（不可驗證的驗收=沒有驗收）
5. ☐ 經濟層執行者：分支判斷都前置了？範例給了？
6. ☐ frontier 執行者：沒有多餘的步驟清單鎖死它？給了 why？
